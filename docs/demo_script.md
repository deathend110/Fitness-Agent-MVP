# RepMind MVP 课堂 Demo 脚本

## 1. Demo 目标

本次课堂 demo 的目标，是在 7 分钟内清楚展示 **RepMind** 这个产品概念，以及它当前版本已经打通的最小闭环：

用户档案 -> 周训练计划 -> 今日日志 -> AI 教练上下文注入 -> AI 建议 -> 一键采纳并写回训练计划。

这次展示不追求“完整健身平台”，而是重点证明两件事：

1. 训练数据和 AI 建议已经能放在同一条流程里。
2. AI 不只是聊天，还能把结构化建议落回训练计划。

## 2. 7 分钟时间分配

- 第 0:00 - 1:00：项目介绍
- 第 1:00 - 4:40：操作流程演示
- 第 4:40 - 5:40：验证结果
- 第 5:40 - 6:40：局限与后续迭代
- 第 6:40 - 7:00：缓冲 / 提问过渡

## 3. 演讲脚本（按分钟段）

### 0:00 - 1:00 项目介绍

大家好，我这次展示的项目叫 **RepMind**。  
这个项目想解决的问题很简单：很多有训练经验的健身用户，平时会把训练计划、体重饮食记录和 AI 建议分散放在不同工具里。结果就是，AI 每次都不知道你最近练了什么、恢复得怎么样，给出的建议也很难直接落回你的计划。

所以我这次做的不是一个“大而全”的健身系统，而是一个本地运行的最小 MVP。它只聚焦一条核心链路：先维护用户档案和周训练计划，再记录今日日志，然后在和 AI 教练对话时自动注入这些上下文。如果 AI 给出结构化训练建议，用户可以一键采纳，把修改直接写回训练计划。

也就是说，这个 MVP 的重点不是“聊天”，而是“让 AI 建议进入真实训练流程”。

### 1:00 - 4:40 操作流程

#### 1:00 - 1:40 第一步：用户档案

现在我先进入“我的档案”页。  
这里可以填写基础信息、训练目标，以及深蹲、卧推、硬拉的 1RM。  
这一步的意义，是给后面的训练计划和 AI 建议提供稳定的个人基线，而不是每次聊天都重新介绍自己。

这里我会快速展示：用户名字、目标、体重，以及三大项 1RM 都已经填写好。

#### 1:40 - 2:20 第二步：周训练计划

接着进入“训练计划”页。  
这里可以维护一周七天的计划，按天配置训练类型和动作内容。  
当前版本支持基于 1RM 百分比或者直接公斤数来维护动作字段。

我这里重点给大家看一个具体动作，比如周一的深蹲。因为后面如果 AI 建议调整，它修改的就是这里已有动作的已有字段。

#### 2:20 - 2:55 第三步：今日日志

然后进入“今日日志”页。  
这里记录当天体重、热量、蛋白质、睡眠、疲劳度、训练完成情况和训练备注。  
我点击保存之后，页面会显示当天最新摘要。

这一步对应的是 README 里的课堂 demo 路径，也就是先完成用户数据录入，再进入 AI 教练环节。

#### 2:55 - 3:40 第四步：AI 教练上下文注入

接着切到“AI 教练”页。  
我会先展开“当前上下文预览”。这里非常关键，因为它能直接证明：系统在发送消息前，已经自动把档案、周计划、近 7 天和近 14 天的日志摘要，以及当日 TDEE 估算注入进去了。

这也是 RepMind 和普通 AI 对话最大的区别。  
普通聊天是用户自己粘贴信息；这里是系统主动组装上下文，让 AI 每次都基于最新状态给建议。

#### 3:40 - 4:10 第五步：发起问题并查看 AI 建议

然后我输入一个训练调整问题，比如“我这周疲劳比较高，周一深蹲要不要降一点强度？”  
发送之后，可以看到聊天消息和加载状态。

如果 AI 回复中包含结构化 JSON 建议，页面下方会出现“建议采纳卡片”。  
这说明系统不只是显示文本，还把 AI 回复解析成了可执行的计划变更。

#### 4:10 - 4:40 第六步：采纳并写回计划

这里我会先简单说明，“忽略”可以把本次建议移除。  
然后重新触发一次带结构化建议的回复，点击“采纳并更新计划”。

之后回到“训练计划”页，确认对应日期、对应动作的字段已经更新。  
这一步就是整个 MVP 最核心的闭环：**AI 建议 -> 用户确认 -> 训练计划写回成功**。

### 4:40 - 5:40 验证结果

这部分我会强调：这个项目不仅能演示，而且已经做了最小验证。

根据 `docs/verification.md`，当前版本已经完成了一条核心成功路径验证，也覆盖了 3 个边界情况和 1 个失败场景。  
自动化验证方面，`npm test` 共 34 个测试全部通过，`npm run build` 也已经成功。

核心成功路径验证的是：今日日志保存后，AI 请求前会构建完整上下文；AI 回复里的正文和结构化建议能够被拆分；用户采纳后，周计划里目标动作的目标字段会被更新，而且不会污染其他动作。

边界情况包括：休息日训练消耗为 0 的上下文构建、新用户没有历史日志时的兜底处理，以及 AI 只返回纯文本时不会错误渲染采纳卡片。  
失败场景则验证了：如果 AI 建议指向不存在的动作，系统会拒绝写回，并保持原计划不变。

### 5:40 - 6:40 局限与后续迭代

最后我也会诚实说明当前版本的边界。

第一，它现在是纯前端本地 MVP，所有数据只保存在浏览器 `localStorage`，没有后端、数据库、账号系统，也没有多端同步。  
第二，AI 建议采纳目前只支持对“已有动作的已有字段”做 `update`，还不支持直接新增动作、删除动作，或者整周自动重排。  
第三，这一版的重点是验证闭环，不是做复杂训练编程。

如果继续迭代，我下一步会优先做三件事：  
第一，补强浏览器端的人工验收和演示稳定性；  
第二，扩展更丰富的建议结构，比如替换动作或新增动作；  
第三，再考虑体重趋势图、数据导入导出、流式输出，或者云同步这类增强功能。

### 6:40 - 7:00 收尾

所以总结一下，RepMind 当前版本已经验证了一个清楚的产品假设：  
**把训练计划、当日状态和 AI 建议放进同一个界面后，AI 就不只是给意见，而是能进入训练计划调整流程。**

我的展示到这里结束，谢谢大家。

## 4. 操作提示 / 演示注意事项

- 课前先确认 `.env` 已配置 `VITE_DEEPSEEK_API_KEY`，并提前启动项目。
- 优先按 README 的 Demo 操作路径演示，不现场自由发挥太多分支。
- 课前准备一套已填写好的档案、周计划和今日日志，避免把时间耗在打字上。
- 在“训练计划”页先记住一个待修改动作，比如周一深蹲，便于采纳后快速对比。
- 进入“AI 教练”页后，先展示“当前上下文预览”，再发送问题，这一步是本项目差异点。
- 如果第一次回复没有带结构化 JSON，不要停留过久，可以换一个更明确的问题继续触发。
- 采纳成功后，第一时间切回“训练计划”页做结果对照，不要只停留在聊天页。
- 若时间紧张，可省略“忽略”按钮演示，把时间留给“采纳并更新计划”。

## 5. 备用说法（如果 AI 接口现场波动怎么办）

### 5.1 接口超时或返回慢

可以这样说：

“AI 教练这一段依赖 DeepSeek 接口，现场网络波动会影响回复速度。但这个项目想验证的核心，不只是大模型能不能回答，而是我们已经把用户档案、训练计划和今日日志自动组织成上下文，并且前端已经具备解析结构化建议和写回计划的能力。”

### 5.2 现场没刷出结构化建议

可以这样说：

“如果这次返回的是纯文本，系统会把它当普通建议显示，而不会错误渲染采纳卡片。这其实也是我们在验证文档里覆盖过的边界情况之一。结构化建议出现时，页面才会进入‘可采纳’模式。”

### 5.3 需要快速切到验证证据

可以这样说：

“为了避免课堂现场把结果完全押在接口波动上，我这里也补充说明一下：当前版本的自动化验证已经覆盖了完整链路。`docs/verification.md` 记录了一条核心成功路径、3 个边界情况和 1 个失败场景，`npm test` 的 34 个测试全部通过，`npm run build` 也成功，所以这个闭环不是只靠现场运气成立的。”

### 5.4 需要解释为什么不演示更多功能

可以这样说：

“这次是课程里的 MVP 展示，所以我刻意把范围收窄到一条最小但完整的链路。当前版本不是要证明‘功能很多’，而是要证明‘这条链路真的能跑通、能验证、能继续迭代’。”
