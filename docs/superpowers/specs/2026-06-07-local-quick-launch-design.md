# RepMind 本地一键快捷启动设计

## 背景

当前 RepMind 的本地启动方式依赖开发者手动执行：

- `npm run dev`
- `npm run dev:backend`
- 或 `npm run dev:all`

这对日常开发没有问题，但对“像桌面应用一样双击启动”这一目标不够友好。用户当前明确需要的是：

- Windows 环境
- 不做安装包
- 不引入 Electron / Tauri / 原生桌面壳
- 依赖本机已经安装好 `Node.js / npm / uv / Python`
- 通过一个入口文件双击即可启动
- 自动打开默认浏览器进入 RepMind
- 启动过程只保留一个启动器窗口，便于观察日志和排错
- 从桌面快捷方式或任意位置点击都能正常工作，不依赖当前工作目录

本设计只覆盖“本地快捷启动器”这一最小包装层，不扩展到桌面安装包或移动端。

## 目标

本次设计目标：

1. 为 RepMind 提供 Windows 本地一键启动入口
2. 支持通过桌面快捷方式或任意目录点击启动
3. 自动启动前端与后端开发服务
4. 自动等待服务就绪后打开默认浏览器
5. 在单个启动器窗口内清晰显示状态与失败原因

## 非目标

本次明确不做：

- Electron / Tauri / pywebview 等桌面壳
- Windows 安装包、`Setup.exe`、MSI
- 独立运行时打包
- 自动安装 `node / npm / uv / python`
- 托盘图标、后台常驻、开机自启
- 生产环境部署或远程后端
- 多平台适配（只覆盖 Windows）

## 用户体验

用户执行路径应收敛为：

1. 双击 `start-repmind.bat`，或双击一个指向该文件的桌面快捷方式
2. 弹出一个 PowerShell 启动窗口
3. 启动窗口依次显示：
   - 当前仓库路径
   - 环境依赖检查结果
   - 清理旧进程结果
   - 前端 / 后端启动中状态
   - 就绪状态
4. 默认浏览器自动打开 `http://127.0.0.1:5173`
5. 启动窗口保留，显示运行提示、日志路径和关闭方式

如果失败，用户应在同一个窗口直接看到：

- 是哪一步失败
- 缺哪个依赖，或哪一个服务没有起来
- 日志文件在哪里

## 方案对比

### 方案 A：PowerShell 启动器 + `.bat` 双击入口

结构：

- `start-repmind.bat`
- `scripts/start-repmind.ps1`

特点：

- `.bat` 只负责双击入口
- 复杂逻辑集中在 PowerShell
- 复用仓库当前已有的 Windows 脚本风格，如 `kill-repmind.ps1`
- 更适合做依赖检查、端口轮询、日志输出和错误提示

优点：

- 与当前仓库风格一致
- 最适合 Windows 本地快捷启动
- 复杂流程可读性和维护性都优于纯批处理

缺点：

- 仍然依赖 PowerShell 执行策略与本机环境
- 不属于真正安装包或独立应用

### 方案 B：纯 `.bat` 启动器

特点：

- 所有逻辑都放在一个批处理文件中

优点：

- 文件数量更少
- 双击入口更“直接”

缺点：

- 错误处理、端口探测、日志跟踪和路径处理都更脆弱
- 后续扩展性差

### 方案 C：Node 启动器 + `.bat` 入口

特点：

- 用一个 Node 脚本做流程控制，再由 `.bat` 调起

优点：

- 逻辑表达能力强
- 未来跨平台扩展更方便

缺点：

- 对当前“Windows 本地快捷启动”目标来说偏重
- 会把简单问题带回 JS 进程编排，收益不高

### 推荐方案

选择方案 A：`PowerShell 启动器 + .bat 双击入口`。

原因：

- 满足当前全部目标
- 对现有仓库侵入最小
- 与已有 `scripts/*.ps1` 生态一致
- 后续如果想接入日志文件、端口切换、启动状态提示，都不需要推翻当前设计

## 文件结构设计

新增两个文件：

### `start-repmind.bat`

职责：

- 作为双击入口
- 基于“脚本自身路径”定位仓库根目录
- 调用 `scripts/start-repmind.ps1`

该文件不承载复杂业务逻辑，只做入口层。

### `scripts/start-repmind.ps1`

职责：

- 统一定位仓库根目录
- 检查依赖
- 清理旧进程
- 启动前后端
- 轮询服务可用性
- 打开默认浏览器
- 打印状态与日志路径
- 在窗口内维持日志输出

## 路径与工作目录规则

启动器必须支持从任意位置触发，包括：

- 仓库根目录双击
- 资源管理器任意路径触发
- 桌面快捷方式触发

因此路径解析必须遵守：

1. 不信任当前 shell 工作目录
2. 始终使用脚本自身路径推导仓库根目录
3. 在执行任何命令前先切换到仓库根目录

这意味着：

- `start-repmind.bat` 通过自身位置定位根目录
- `scripts/start-repmind.ps1` 也独立通过自身路径反推根目录

即使 `.bat` 被移动为快捷方式入口，只要目标仍指向仓库内原文件，脚本就能正常工作。

## 启动流程设计

启动器统一按以下顺序执行：

### 1. 进入仓库根目录

- 基于脚本路径切换到项目根目录
- 验证 `package.json`、`backend/`、`scripts/` 等关键路径存在

若仓库结构不完整，直接失败并提示当前脚本不在有效工作区。

### 2. 检查依赖

必须检查：

- `node`
- `npm`
- `uv`

检查原则：

- 任一缺失即失败
- 在窗口中明确输出缺失项
- 不做自动安装

说明：

- `uv run python ...` 已隐含依赖可用 Python 运行时，因此优先检查 `uv` 即可；是否单独检查 `python` 可作为辅助提示，但不作为主执行入口。

### 3. 清理旧进程

- 复用现有 `scripts/kill-repmind.ps1`
- 清理可能残留的本项目本地前后端进程

目的：

- 避免 5173 / 8000 端口被旧实例占用
- 避免双击多次后前后端重叠启动

### 4. 创建本轮日志目录或日志文件

建议在仓库内固定目录保存启动日志，例如：

- `tests/reports/local-launch/`

每次启动写入两份日志：

- 前端日志
- 后端日志

日志文件名应带时间戳，避免覆盖上一轮结果。

### 5. 启动后端

命令：

```powershell
uv run python -m backend.run_dev_server
```

要求：

- 后台进程方式启动
- stdout/stderr 重定向到后端日志文件

### 6. 启动前端

命令：

```powershell
npm run dev -- --host 127.0.0.1 --port 5173 --strictPort
```

要求：

- 后台进程方式启动
- stdout/stderr 重定向到前端日志文件

使用固定本地地址的目的：

- 保持浏览器自动打开地址稳定
- 避免 `vite` 因环境差异启到其他 host 或 fallback 端口

### 7. 等待服务就绪

启动器应轮询：

- `http://127.0.0.1:8000/api/health`
- `http://127.0.0.1:5173`

规则：

- 仅当两者都可访问时视为启动成功
- 若超时未就绪，直接失败
- 失败时打印哪个服务未就绪，并给出日志路径

### 8. 打开默认浏览器

在服务就绪后自动打开：

```text
http://127.0.0.1:5173
```

浏览器选择策略：

- 使用系统默认浏览器
- 不固定到 Chrome / Edge

### 9. 保持启动器窗口

成功后，当前窗口不退出，而是显示：

- 启动成功
- 前端地址
- 后端健康检查地址
- 日志文件路径
- 关闭方式提示

之后继续输出或尾随日志，保证出现运行时错误时用户能在同一窗口看到。

## 日志与可观察性

为保证排错效率，启动器必须打印：

- 当前仓库根目录
- 依赖检查结果
- 启动命令摘要
- 前端日志路径
- 后端日志路径
- 服务就绪状态

日志设计原则：

- 不把全部逻辑藏在静默后台
- 出错时用户无需再猜是哪一侧失败
- 日志路径稳定且容易找到

## 失败处理

以下情况必须显式失败：

- 仓库根目录解析失败
- `node / npm / uv` 缺失
- 旧进程清理脚本异常
- 后端超时未启动
- 前端超时未启动
- 打开浏览器前任一服务异常退出

失败输出至少包含：

- 失败步骤名称
- 失败原因概述
- 建议查看的日志文件路径

本次不做自动重试与自动恢复，只做明确失败提示。

## 关闭与清理策略

本次不设计复杂的托盘关闭或进程回收编排。

关闭方式保持简单：

- 用户关闭当前启动器窗口不会自动保证终止所有子进程
- 用户如需手动清理，可继续使用现有：

```powershell
npm run stop:all
```

这是当前版本的有意边界，避免本次将“快捷启动器”扩展成完整桌面进程管理器。

## 测试与验收标准

最小验收标准如下：

1. 从仓库根目录双击 `start-repmind.bat` 可正常启动
2. 从桌面快捷方式点击 `start-repmind.bat` 也可正常启动
3. 启动器会自动切换到仓库根目录
4. 当前窗口只保留一个启动器窗口
5. 启动成功后默认浏览器自动打开 `http://127.0.0.1:5173`
6. 前后端地址和日志路径会被清晰打印
7. 缺失依赖时能明确报错并停止
8. 前端或后端启动失败时能明确提示失败步骤和日志路径

## 实现边界总结

本设计只解决：

- Windows 本地一键启动
- 任意位置触发
- 单窗口显示启动状态
- 自动打开默认浏览器

本设计不解决：

- 安装包分发
- 原生桌面壳
- 独立运行时
- 跨平台启动器
- 完整后台生命周期管理

## 结论

本次采用：

- `start-repmind.bat` 作为双击入口
- `scripts/start-repmind.ps1` 作为主启动器

以最小代价把当前“开发命令启动”提升为“本地一键快捷启动”，同时保持：

- 低侵入
- 易维护
- 可排错
- 与现有仓库脚本风格一致
