# RepMind 发布前终极测试门禁设计

## 1. 背景与目标

RepMind 当前已经具备前端单测、后端 pytest 和若干 Playwright E2E，但这些测试仍然更偏“功能回归”，还不是一套真正意义上的发布门禁。项目在发布前需要一套可重复执行、可定位失败原因、能够覆盖真实用户乱操作风险的总测试方案。

本设计文档定义一套面向 `CI 可重复执行` 的发布前终极测试门禁。目标不是单纯增加测试数量，而是把“项目是否进得起不同用户的随意操作”拆解成若干可自动执行的风险层，并统一收束到一个发布门禁入口。

## 2. 成功标准

这套发布门禁达成后，项目发布前应满足以下条件：

1. 存在一个统一的总入口脚本，可以一键执行完整发布门禁。
2. 前端、后端、浏览器真实交互、真实 AI 外部依赖都被纳入统一门禁。
3. 任一阶段失败时，可以快速定位是环境、前端、后端、E2E、真实 provider 还是高强度扰动测试出错。
4. 高风险用户行为，包括非法输入、频繁切换、刷新恢复、重复提交、重复采纳、重复拖拽等，都有对应的自动化覆盖。
5. 整套门禁可在高强度模式下运行 30 分钟以上，但结果必须稳定、可复现，不依赖临时人工操作。

## 3. 非目标

本设计不追求以下目标：

1. 不要求把所有用户行为都自动化覆盖。
2. 不要求真实 AI 测试覆盖所有模型 provider，只要求至少一条真实外部链路门禁。
3. 不在本阶段引入云端 CI 平台绑定实现细节，例如 GitHub Actions YAML 或其他平台配置。
4. 不把课程展示类的纯视觉细节全部纳入发布阻断条件，重点仍然是闭环正确性、持久化稳定性和 AI 链路可用性。

## 4. 风险模型

发布前需要重点防御的风险分为六类：

1. 环境与文档风险：开发者按文档无法启动、缺环境变量、缺浏览器内核、脚本失效。
2. 前端逻辑回归：视图模型、输入 guardrail、状态同步、计划写回、建议卡逻辑退化。
3. 后端服务回归：API、provider runtime、tool loop、周期计划、迁移、记忆、文件链路异常。
4. 浏览器真实交互回归：单测通过，但真实页面操作时状态错乱、DOM 交互失效、刷新恢复异常。
5. 真实 AI 外部依赖失效：API Key、模型名、超时、响应结构波动导致实际问答链路失败。
6. 用户乱操作压力风险：乱输、乱点、乱切页、乱刷新、重复提交导致页面或数据进入错误状态。

发布门禁需要逐层覆盖这六类风险，而不是用一条超大脚本模糊兜底。

## 5. 总体方案

推荐方案为“分层门禁，统一总入口”。

总入口负责统一调度六个阶段，但每个阶段都保留独立职责、独立日志和独立失败定位能力。这样既满足一键执行，也能在失败时快速判断是哪一层出了问题。

### 5.1 总门禁阶段

1. 阶段 0：环境与规范门禁
2. 阶段 1：前端单测与构建门禁
3. 阶段 2：后端 pytest 门禁
4. 阶段 3：浏览器核心回归门禁
5. 阶段 4：真实 AI 冒烟门禁
6. 阶段 5：高强度扰动门禁

### 5.2 统一入口形式

建议提供一个统一脚本，例如：

- `scripts/run-release-gate.ps1`

该脚本串行执行所有阶段，并输出统一测试报告。之所以优先使用 PowerShell，而不是 Bash 或单独 Python 编排，是因为当前项目开发环境和已有脚本已经明显偏向 Windows 本地工作流，直接集成成本最低。

## 6. 各阶段设计

### 6.1 阶段 0：环境与规范门禁

目标是阻断“代码本身没问题，但项目根本起不来”的发布风险。

必须覆盖：

1. `npm install` 可执行。
2. `uv sync` 可执行。
3. `uv run python -m playwright install chromium` 可执行。
4. `.env.example` 与 `backend/.env.example` 包含发布测试所需关键字段。
5. `README.md` 和 `ARCHITECTURE.md` 存在，且包含启动、测试、AI 配置相关信息。
6. 总门禁依赖的脚本、目录和必要测试文件存在。

建议补一个轻量检查脚本，例如：

- `scripts/check-release-env.py`

这个脚本不负责安装依赖，只负责检查模板文件、必要环境变量名、文档关键段、测试入口文件是否存在。

### 6.2 阶段 1：前端单测与构建门禁

目标是阻断前端逻辑回归和构建失效。

必须覆盖：

1. `npm test`
2. `npm run build`

当前项目已有大量前端测试，已经覆盖：

1. 档案页输入约束与摘要逻辑
2. 训练计划头部、布局、动作编辑、拖拽排序、周期切换
3. 今日日志录入与指标视图
4. AI 教练请求、建议卡、历史恢复、流式状态
5. 数值 guardrail、Markdown 渲染、模型配置等辅助能力

因此该阶段重点不是新增大量单测，而是把全量前端测试正式纳入发布门禁。

### 6.3 阶段 2：后端 pytest 门禁

目标是阻断后端服务层、数据层和 AI 编排层回归。

必须覆盖：

1. `uv run pytest backend/tests -q`

当前后端测试已经覆盖：

1. 配置加载
2. profile / weekly plan / cycle / daily log API
3. chat session / context manager / tool loop
4. provider runtime 与 Gemini/OpenAI-compatible 适配
5. memory / migrate / files / metrics / model config

该阶段的关键是把它定义为正式发布门禁中的阻断项，而不是日常参考项。

### 6.4 阶段 3：浏览器核心回归门禁

目标是验证核心业务闭环在真实页面中可正常工作。

必须包含五条核心链路：

1. 档案填写并持久化。
2. 训练计划编辑、拖拽排序、刷新恢复。
3. 今日日志录入、非法输入拦截、刷新恢复。
4. AI 教练普通问答、会话切页回页、历史恢复。
5. AI 建议卡出现、采纳并写回训练计划。

优先复用现有稳定脚本，例如：

1. `tests/e2e/coach_browser_smoke.py`
2. `tests/e2e/plan_drag_sort.py`
3. `tests/e2e/coach_commit_full_flow.py`
4. `tests/e2e/coach_session_history.py`
5. `tests/e2e/coach_model_config_flow.py`

必须新增一条完整发布主链路脚本：

- `tests/e2e/release_core_journey.py`

该脚本需要把 `Profile -> Plan -> Today -> Coach -> Adopt -> Plan` 串成一条完整可重复验证的链路，作为发布门禁的业务代表样本。

### 6.5 阶段 4：真实 AI 冒烟门禁

目标是阻断“mock 全绿，真实模型链路却不可用”的风险。

必须新增：

- `tests/e2e/coach_real_provider_smoke.py`

测试要求：

1. 使用真实 provider 配置启动后端。
2. 通过前端或页面级 E2E 发起一次真实问答。
3. 成功收到真实回复。
4. 若当前 provider 稳定支持结构化建议，则可以扩展为建议卡链路验证；若模型输出稳定性不足，则本阶段只以“真实回复成功”作为硬门槛。

设计约束：

1. 断言不能写死具体回复内容。
2. 允许基于关键词、非空文本、消息落库等方式做稳定断言。
3. 允许更长超时与有限重试，但失败仍应视为发布阻断。

### 6.6 阶段 5：高强度扰动门禁

目标是逼近“不同用户随意操作”的真实风险。

必须覆盖四类扰动：

1. 非法输入风暴
2. 高频切换
3. 持久化恢复
4. 重复动作

建议新增脚本：

1. `tests/e2e/profile_input_fuzz.py`
2. `tests/e2e/today_log_fuzz.py`
3. `tests/e2e/plan_mutation_stress.py`
4. `tests/e2e/navigation_recovery_stress.py`

具体要求如下：

#### 非法输入风暴

验证：

1. 超大值、负值、边界小数、清空、半输入态不会把非法值落库。
2. 字段级错误提示能稳定出现。
3. 刷新后页面不会恢复到错误状态。

#### 高频切换

验证：

1. 多次快速切换 tab 不会造成编辑态、加载态或消息态错乱。
2. 弹窗多次打开/关闭后仍可继续提交。
3. 编辑过程中切页再回来，不会破坏关键状态。

#### 持久化恢复

验证：

1. 页面刷新后档案、计划、日志、会话状态恢复一致。
2. 关键操作后的 localStorage 与后端回流状态一致。
3. 多次刷新后不会出现重复消息、重复建议卡或顺序错乱。

#### 重复动作

验证：

1. 连续发送消息不会造成重复提交失控。
2. 连续采纳建议不会重复写回。
3. 连续拖拽、连续切换计划来源、连续生成下一周不会把状态推入非法分支。

扰动测试必须是“固定动作序列或固定随机种子”的可复现脚本，不允许使用纯随机且不可重放的实现。

## 7. 报告与产物

总门禁脚本执行后，需要在统一目录下输出测试产物，建议目录为：

- `tests/reports/release-gate/`

至少保存：

1. 每阶段开始与结束时间
2. 每阶段命令输出
3. 前端测试结果日志
4. 后端 pytest 结果日志
5. E2E 截图、失败页面 HTML、控制台日志
6. 真实 AI 冒烟日志
7. 一份最终 `summary.json`

`summary.json` 应至少包含：

1. 执行时间
2. 各阶段状态
3. 各阶段耗时
4. 失败阶段与失败命令
5. 最终总结果

## 8. 通过标准

正式发布必须满足：

1. 六个阶段全部通过。
2. 任一阶段失败即阻断发布。
3. 真实 AI 冒烟失败视为发布阻断。
4. 高强度扰动脚本必须稳定，不允许默认忽略 flaky。

额外规则：

1. 如果某条 E2E 长期 flaky，应先移出正式门禁并进入修复清单，待稳定后再纳回。
2. 正式门禁内的脚本必须支持无人工干预执行。
3. 所有新增脚本都应优先使用现有 E2E helper、自举前端和统一 mock 能力，避免重复造轮子。

## 9. 分阶段实施顺序

### 9.1 第一批

优先完成：

1. 总门禁脚本
2. 阶段 0、1、2 接通
3. 现有稳定 E2E 纳入阶段 3
4. 新增 `release_core_journey.py`
5. 新增 `coach_real_provider_smoke.py`

达成后，项目已经拥有可工作的发布门禁主骨架。

### 9.2 第二批

补强乱操作能力：

1. `profile_input_fuzz.py`
2. `today_log_fuzz.py`
3. `plan_mutation_stress.py`
4. `navigation_recovery_stress.py`

这一批直接对应“不同用户随意操作”的高风险场景。

### 9.3 第三批

提升可维护性：

1. 统一报告输出
2. 失败归档规范
3. 阶段耗时分析
4. 真实 AI 冒烟超时与重试策略
5. flaky 用例治理规则

## 10. 推荐实现目录

建议本次发布门禁相关实现最终落在以下位置：

1. `scripts/run-release-gate.ps1`
2. `scripts/check-release-env.py`
3. `tests/e2e/release_core_journey.py`
4. `tests/e2e/coach_real_provider_smoke.py`
5. `tests/e2e/profile_input_fuzz.py`
6. `tests/e2e/today_log_fuzz.py`
7. `tests/e2e/plan_mutation_stress.py`
8. `tests/e2e/navigation_recovery_stress.py`
9. `tests/reports/release-gate/`

## 11. 设计结论

RepMind 的发布前终极测试，不应继续停留在“前端单测 + 后端 pytest + 零散 smoke”的层级，而应升级为一套六阶段发布门禁。该门禁通过分层设计覆盖环境、代码、浏览器、真实 AI 和高强度扰动风险，并通过统一总入口实现真正的发布阻断能力。

这套方案的关键价值不是测试数量，而是把发布风险结构化、自动化、可追踪化。只有这样，它才足以承担“项目发布前大测”的角色。
