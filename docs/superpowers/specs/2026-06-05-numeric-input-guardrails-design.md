# 数值输入边界约束设计

## 背景

当前工作区里，“我的档案”“今日日志”“训练计划”三个页面都存在数值输入，但除个别字段外，大多数输入只做了字符串转数字，没有统一的业务边界约束。

这会导致以下问题：

- 用户可以输入明显违背生活常识或物理规律的值，例如 `9999 kg`、`-1 kg`、`100000 kg`
- 前端不同页面的限制风格不一致，只有少数字段带 `min / max`
- 即使浏览器原生 `type="number"` 生效，也不能保证所有入口、受控状态和程序态写入都得到一致约束
- 后续如果继续扩展计划页、周期页和日志页，范围规则容易继续分散

本设计的目标是在不引入后端新架构的前提下，为三个页面建立统一、可复用、可测试的数值输入边界约束。

## 目标

- 为“我的档案”“今日日志”“训练计划”三个页面的数值字段建立统一范围规则
- 输入阶段就限制越界值，并同步给出错误提示
- 保存前再做同源校验，防止异常值通过其他路径写入业务数据
- 保持空值可表达，避免破坏现有“可留空”的 MVP 录入体验
- 让规则可复用到手动训练计划、周期计划、自定义力量周期计划
- 同步更新 `README.md` 和 `ARCHITECTURE.md`，明确新增的输入约束层

## 非目标

- 不在本次工作中引入后端数据库级约束
- 不重做全部表单 UI
- 不改动与数值边界无关的业务字段
- 不为文本字段增加复杂校验
- 不处理历史数据库脏数据的批量迁移

## 用户确认的交互策略

用户已确认采用以下策略：

- 超出范围时，在输入阶段就限制
- 同步显示错误提示
- 不能只在点击保存时才报错

因此本次实现不会采用“先允许录入、保存时统一拦截”的方案，而是采用“输入阶段约束 + 保存前复核”的双层策略。

## 方案比较

### 方案 A：只依赖原生 `min / max / step`

优点：

- 改动最小
- 浏览器端实现直接

缺点：

- 各页面规则仍然分散
- 受控输入和程序态赋值不能完全依赖原生行为
- 错误提示不可控，跨组件体验不一致

### 方案 B：建立统一范围表，并让输入层与保存层复用

优点：

- 规则集中，便于维护
- 输入层和数据层使用同一套边界来源，避免分叉
- 测试可以直接覆盖规则映射和越界处理
- 适合当前项目继续迭代

缺点：

- 需要增加一层共享规则和少量表单适配逻辑

### 方案 C：每个页面自行维护本地范围判断

优点：

- 初始编码速度快

缺点：

- 同类字段容易出现不同范围
- 后续修改成本高
- 不适合周期计划和动作编辑这种多入口场景

### 结论

采用方案 B。

## 架构设计

### 1. 新增统一数值规则层

新增一个前端共享规则模块，职责如下：

- 定义每个数值字段的范围配置
- 输出表单层可直接使用的 `min / max / step`
- 输出统一错误提示文案
- 提供“空值放行、填写后必须校验”的通用判定函数
- 提供对字符串输入的标准化工具，保证受控输入稳定

该模块只负责“规则表达”和“规则判定”，不直接依赖 React 组件。

### 2. 页面表单层接线

三个页面的数值输入组件统一接线到共享规则层：

- 输入框渲染时带上对应 `min / max / step`
- `onChange` 阶段判断是否越界
- 越界时保留用户可理解的提示状态，但不把非法数值写入最终业务数据
- 页面或组件在需要时显示字段级错误提示

### 3. 保存前复核

虽然主策略是输入阶段限制，但保存前仍需再走一次规则层复核：

- `profile` 草稿转业务对象前复核
- `today log` 草稿写回前复核
- `exercise` 草稿保存前复核
- `cycle` / `custom strength` payload 构建前复核

这样可以防止：

- 受控状态被其他逻辑直接改写
- 历史测试或未来功能绕过输入层
- 程序态数据合成时写入异常值

### 4. 错误展示原则

错误展示统一遵循以下原则：

- 错误提示尽量贴近字段，不只在页面顶部统一提示
- 已有字段提示区域优先复用，不额外堆积大块 UI
- 错误文案包含字段名、单位和范围信息
- 空值不报错
- 文本字段和选择字段不受本次规则影响

## 数据流变化

### 我的档案页

1. 用户输入年龄、身高、体重、腰围、目标体重或 1RM
2. 表单层读取共享数值规则
3. 越界时立即反馈错误并阻止非法业务值继续传播
4. `draftToProfile()` 在输出前再次复核
5. 合法值才写入页面状态并同步后端

### 今日日志页

1. 用户输入体重、热量、蛋白质、睡眠、步数、TDEE、疲劳度
2. 字段组配置从共享规则层读取显示约束
3. 输入阶段即时提示
4. `normalizeTodayLogEntry()` / `buildTodayLogPayload()` 写回前再次复核
5. 合法值才进入 `dailyLog`

### 训练计划页

1. 用户编辑周数、动作百分比、固定重量、组数、次数、RPE
2. 动作编辑器与周期设置表单统一接共享规则
3. 越界值即时提示，不能静默写入
4. `buildExerciseSavePayload()`、周期 payload builder 在提交前复核
5. 合法值才进入 `weeklyPlan` 或周期创建请求

## 测试设计

本次实现遵守 TDD，至少覆盖以下验证。

### 核心成功路径

- 档案页输入合理体重、身高、1RM，可以正常保存
- 今日日志输入合理体重、睡眠、疲劳度，可以正常保存
- 训练计划页输入合理动作重量、组数、次数、RPE，可以正常保存

### 边界情况

- 正好等于最小值时允许保存
- 正好等于最大值时允许保存
- 空值保持允许，不因为新增约束导致无法清空字段

### 失败场景

- 输入 `9999 kg` 体重时立即报错
- 输入 `-1 kg` 卧推 1RM 时立即报错
- 输入 `100000 kg` 固定训练重量时立即报错

### 回归验证

- 现有 `fatigue 1-5` 和 `RPE 0-10` 规则保持成立
- 周期计划创建 payload 不会因新增校验破坏既有正常输入
- 自定义力量周期计划的周数调整仍能稳定扩缩周数组

## 风险与处理

### 风险 1：受控输入过早拦截导致输入体验卡顿

处理：

- 允许空串状态
- 优先在字符串层判断，再决定是否转换为业务数值
- 避免把用户输入中的中间态直接当成最终业务值

### 风险 2：已有工具函数直接把异常值转成数字

处理：

- 在工具函数层增加同源复核
- 让表单层和工具层都复用同一份规则

### 风险 3：规则散落在页面和工具层两份实现

处理：

- 只保留一份共享规则定义
- 页面和工具只调用规则，不重复硬编码范围

### 风险 4：README 与架构文档滞后

处理：

- 本次任务将文档同步列为必做项，不作为可选收尾

## 验收标准

- 三个页面所有已确认的数值字段都具备明确的范围限制
- 用户无法在输入阶段把明显异常数值写成有效业务状态
- 每个受限字段都有可理解的范围提示或错误提示
- 保存前有同源复核，避免绕过 UI 写入异常值
- 相关测试覆盖成功、边界和失败路径
- `README.md` 和 `ARCHITECTURE.md` 已同步说明该约束层
