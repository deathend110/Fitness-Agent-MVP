# V2.5 单日训练计划卡采纳设计

> 日期：2026-06-01
> 范围：AI 教练生成“单日训练计划卡”后，用户确认即可把该日训练类型、动作和动作详情直接写回训练计划页
> 上游背景：[V2.3 开发完成总结](<../V2.3/V2.3 开发完成总结.md>) · [V2.4 验收记录](<../V2.4/V2.4 验收记录.md>)

## 1. 背景与现状

当前仓库里已经存在“AI 建议卡 -> 用户确认 -> 写回训练计划”的基础链路：

- 前端 `CoachTab.handleAdoptSuggestion()` 会在用户点击“采纳并更新计划”后调用：
  - 新版：`POST /api/tools/plan/commit`
  - 旧版兼容：`POST /api/weekly-plan/adopt`
- 后端 `backend/api/tools.py` 已支持按 `proposalId` 提交 proposal
- 后端 `backend/agent/adopt_plan.py` 已支持将 proposal 写回 `weekly_plan`
- 前端成功后会通过 `onWeeklyPlanChange(adoptResult.plan)` 更新训练计划页

这说明“确认后完全不能写回”并不是当前系统的真实状态。现有链路在以下场景已经成立：

- 修改已存在训练日里的已存在动作
- 修改动作的已有字段，例如：
  - `pct`
  - `kg`
  - `sets`
  - `reps`
  - `rpe`
  - `note`

同时，现有测试也已经覆盖了这条链路：

- `backend/tests/test_plan_tools.py`
- `backend/tests/test_adopt_plan.py`
- `tests/e2e/coach_browser_smoke.py`

## 2. 当前问题

本次用户要的不是“再做一个更新某个字段的按钮”，而是：

> 给出的训练计划卡我确认后，agent 能直接把对应的动作和相关信息填入训练计划页

按产品语义理解，这更接近：

- AI 生成一张“某一天的训练计划卡”
- 用户确认后，直接把该日训练计划写回
- 写回内容不再局限于某个动作的单字段修改，而是可能包括：
  - 修改训练日类型
  - 新增动作
  - 删除动作
  - 调整动作顺序
  - 直接填写动作的完整结构化信息

当前实现无法满足这一点，核心原因有 4 个：

### 2.1 后端只支持 `action="update"`

`backend/agent/adopt_plan.py` 中的 `adopt_plan_change()` 当前明确只接受：

```text
action == "update"
```

其余动作会被直接拒绝：

- `add`
- `remove`
- `replace`

### 2.2 写回目标只能是“已存在动作的已存在字段”

当前 proposal 只能按：

- `day`
- `exerciseName`
- `field`
- `newValue`

这种“字段 patch”语义来写回。

这意味着它不能表达：

- 新增一条完整动作
- 删除一条动作
- 整体替换某一天的动作列表
- 同时修改 `type + exercises[]`

### 2.3 前端展示层与后端能力不一致

前端 `src/utils/adoptCard.js` 已经预留了：

- `add`
- `remove`
- `update`

三类卡片文案。

但后端真正 commit 时只支持 `update`，所以用户从卡片上看到“新增/删除/整日调整”的语义，并不代表系统真的能写回。

### 2.4 现有 proposal 粒度太细，不适合“单日训练计划卡”

当前 proposal 更像：

- “把周一深蹲 `sets` 从 4 改为 3”

而不是：

- “把周一改成恢复型腿日，并新增 3 个动作，保留 1 个动作，删除 1 个动作”

用户要的 `B` 方案本质上是后者。

## 3. 目标

本次 V2.5 目标是实现 `B` 方案：

- AI 教练可以生成“单日训练计划卡”
- 卡片至少包含：
  - 目标日期
  - 训练日类型
  - 动作列表
  - 每个动作的关键训练信息
  - 一段简短调整说明
- 用户点击确认后：
  - 对应日期的训练计划被直接写回
  - 训练计划页立刻显示最新结果
  - 后端 SQLite 持久化同步更新

本次写回目标是“整日计划替换”，而不是“局部字段 patch”。

## 4. 非目标

本次不做：

- 一次性生成并采纳整周计划
- 自动无确认写回
- 对单日计划做差异可视化编辑器
- proposal 历史版本管理
- undo/rollback 栈
- 多人协同冲突处理

## 5. 方案选择

本次考虑 3 条路径：

### 方案 A：继续沿用旧 `changes[]`，只扩展 `add/remove`

做法：

- 继续使用当前 proposal 结构
- 在 `changes[]` 中支持：
  - `add`
  - `remove`
  - `update`

优点：

- 复用现有 `/api/tools/plan/commit`
- 对现有前端采纳按钮改动较小

缺点：

- `changes[]` 会变得很复杂
- 很难稳定表达动作顺序、day type、完整动作结构
- AI 容易输出不完整 patch
- 对“单日训练计划卡”的表达仍然拧巴

结论：

- 不推荐作为 V2.5 主方案

### 方案 B：新增“整日计划 proposal”能力，与旧字段 patch proposal 并存

做法：

- proposal 增加 `kind`
- 新增 `day_plan_replace` 类型
- commit 时按 `kind` 分流：
  - 旧 `field_changes`
  - 新 `day_plan_replace`

优点：

- 最贴合“确认一张单日训练计划卡后直接写回”的产品语义
- 与现有 proposal/commit 基础设施兼容
- 可以保留旧卡片链路，不破坏现有测试
- 单日计划对象更容易校验、归一化和展示

缺点：

- 需要扩展 proposal 存储结构、tool schema 和前端卡片展示
- 需要增加整日计划校验器

结论：

- 推荐本次采用

### 方案 C：跳过 proposal，AI 直接调用写库工具

做法：

- 允许模型直接执行写库工具
- 不再经过用户确认 proposal

优点：

- 技术链路短

缺点：

- 违反当前产品与架构边界
- 不可接受，存在误写库风险

结论：

- 明确不采用

## 6. 推荐方案

推荐采用 **方案 B：保留旧 proposal/commit 框架，新增“单日训练计划卡” proposal 类型**。

核心思路：

- 后端继续只允许“proposal -> 用户确认 -> commit”
- proposal 从“只会字段 patch”升级为“支持 proposal kind”
- 新增一种能描述“整日训练计划”的 proposal
- 前端采纳入口和 commit 接口继续复用，不改用户操作心智

最终效果：

- 旧的轻量字段调整卡仍可正常使用
- 新的整日训练计划卡也可被确认并写回
- 训练计划页继续只消费 `weeklyPlan` 状态，无需知道 proposal 内部细节

## 7. 数据结构设计

### 7.1 Proposal 顶层结构

当前 proposal 只有“卡片内容 + proposalId”的轻量语义。

V2.5 后 proposal 需要显式区分种类：

```json
{
  "proposalId": "proposal-uuid",
  "kind": "day_plan_replace",
  "day": "Monday",
  "summary": "将周一改为恢复型腿日，降低总疲劳。",
  "status": "pending"
}
```

其中：

- `kind`
  - `field_changes`：旧版字段修改 proposal
  - `day_plan_replace`：新版整日计划 proposal

### 7.2 新版单日训练计划 proposal 卡片

新增 `dayPlan` 载荷：

```json
{
  "proposalId": "proposal-uuid",
  "kind": "day_plan_replace",
  "day": "Monday",
  "summary": "将周一改为恢复型腿日，降低总疲劳。",
  "dayPlan": {
    "type": "腿日",
    "exercises": [
      {
        "id": "proposal-monday-squat",
        "name": "深蹲",
        "tier": "main",
        "template": {
          "loadMode": "percentage",
          "ref1RM": "squat",
          "setType": "straight",
          "sets": 3,
          "repsText": "5"
        },
        "instance": {
          "pct": 0.7,
          "kg": null,
          "rpe": 7,
          "note": "恢复周主项"
        },
        "ref1RM": "squat",
        "pct": 0.7,
        "kg": null,
        "sets": 3,
        "reps": 5,
        "rpe": 7,
        "note": "恢复周主项"
      }
    ]
  },
  "status": "pending"
}
```

设计要点：

- `dayPlan` 必须是完整单日结构
- 进入 commit 前先做结构归一化
- commit 后直接覆盖目标日的 `type + exercises`

### 7.3 旧版 proposal 兼容

旧 proposal 继续保留：

```json
{
  "proposalId": "proposal-uuid",
  "kind": "field_changes",
  "day": "Monday",
  "summary": "把深蹲总量降一组。",
  "changes": [
    {
      "action": "update",
      "exerciseName": "深蹲",
      "field": "sets",
      "oldValue": 4,
      "newValue": 3
    }
  ],
  "status": "pending"
}
```

这样可以保证：

- 现有测试不必整体推倒重来
- 旧消息历史中的 suggestion/proposal 仍可恢复显示

## 8. 后端设计

### 8.1 `backend/agent/adopt_plan.py`

当前模块职责偏“字段 patch commit”。

V2.5 后建议拆成两类能力：

#### 8.1.1 旧能力保留

- `adopt_plan_change()`
- `build_plan_change_proposal()`
- `commit_validated_plan_change()` 中 `field_changes` 分支

#### 8.1.2 新增整日计划能力

建议新增：

- `validate_day_plan_payload(day, day_plan)`
- `normalize_day_plan_payload(day_plan)`
- `build_day_plan_replace_proposal(...)`
- `commit_day_plan_replace_proposal(...)`

其中：

- `validate_day_plan_payload()`
  - 验证 day 合法
  - 验证 `type` 合法
  - 验证 `exercises` 为数组
  - 验证每条动作至少具备 `name`
  - 验证 `sets/reps/rpe/pct/kg` 的数据范围
- `normalize_day_plan_payload()`
  - 复用当前前端/后端已有的 normalize 规则
  - 输出与训练计划页一致的动作结构

### 8.2 Proposal Store 结构升级

当前 `_proposal_store` 只保存旧 proposal。

V2.5 需要把 `PlanChangeProposal` 升级为更泛化的 proposal 结构，至少包含：

- `proposal_id`
- `session_id`
- `kind`
- `day`
- `summary`
- `card`
- `validation`
- `status`
- `payload`

其中：

- 旧 proposal 的 `payload` 为 `changes`
- 新 proposal 的 `payload` 为 `dayPlan`

### 8.3 `backend/api/tools.py`

现有接口继续保留：

- `POST /api/tools/plan/propose`
- `POST /api/tools/plan/commit`

但内部逻辑需要升级。

#### `POST /api/tools/plan/propose`

建议扩展请求体，支持两种模式：

- 旧模式：`changes[]`
- 新模式：`dayPlan`

为避免一个接口既收 patch 又收完整计划导致歧义，推荐在请求体中显式增加：

```json
{
  "kind": "day_plan_replace"
}
```

#### `POST /api/tools/plan/commit`

请求体仍保持：

```json
{
  "proposalId": "proposal-uuid"
}
```

好处：

- 前端采纳按钮不必改接口形状
- commit 时只根据 proposal store 中的 `kind` 分流

### 8.4 Tool Calling 能力

当前模型侧只注册了：

- `propose_plan_change`

V2.5 推荐新增一个更语义化的 tool，而不是把旧 tool 搞成同时支持两种复杂输入：

- `propose_day_plan_replace`

建议参数：

```json
{
  "day": "Monday",
  "summary": "恢复型腿日",
  "dayPlan": {
    "type": "腿日",
    "exercises": [ ... ]
  }
}
```

推荐新增独立 tool 的原因：

- 对模型提示更清晰
- 减少 patch proposal 与整日计划 proposal 的 schema 干扰
- 便于测试和观测 tool call 命中情况

## 9. 前端设计

### 9.1 `CoachTab.jsx`

当前 `handleAdoptSuggestion()` 已经做了 commit 分流：

- 有 `proposalId` -> `commitPlanChange`
- 无 `proposalId` -> `adoptWeeklyPlanChange`

这层逻辑总体可复用。

本次主要改动不在按钮事件，而在 commit 成功后的状态合并策略：

#### 现状

```javascript
onWeeklyPlanChange(adoptResult.plan)
```

#### 风险

后端返回的 `plan` 只有标准周计划 7 天结构，不包含前端可能保留的扩展元数据，例如：

- `weekMeta`
- 未来本地 UI 临时字段

#### 推荐改法

新增一个小型 merge helper，例如：

```javascript
mergeCommittedWeeklyPlan(currentPlan, committedPlan)
```

策略：

- 以当前前端状态为基底保留顶层扩展元数据
- 用后端返回的 7 天 canonical 数据覆盖对应日期

这样能避免“确认写回成功，但训练计划页某些本地周信息被冲掉”的体验问题。

### 9.2 `buildAdoptCardModel()`

当前卡片模型只会把 `changes[]` 渲染成字段变化列表。

V2.5 后需要支持两类卡片：

- `field_changes`
- `day_plan_replace`

对于 `day_plan_replace`：

- 顶部展示：
  - 目标日期
  - summary
  - 训练日类型
- 中部展示：
  - 动作列表预览
  - 每个动作的组数、次数、负重模式、RPE、备注

这张卡本质上不再是“字段 diff 列表”，而是“单日计划预览卡”。

### 9.3 `AdoptCard.jsx`

当前组件默认是“变化清单型卡片”。

V2.5 后建议改成多形态组件：

- `variant: "changes"`
- `variant: "dayPlan"`

其中 `dayPlan` 形态需要：

- 更强的结构化展示
- 但仍保持 MVP 级别，不做复杂可编辑 UI

## 10. 训练计划页写回行为

训练计划页本身不需要知道“这次 commit 来自 AI 还是来自手工编辑”。

它只需要消费更新后的 `weeklyPlan`。

因此：

- `PlanTab.jsx` 主逻辑不需要为 proposal 增加特殊分支
- 只要 `App.jsx` 中的 `weeklyPlan` 被正确更新，训练计划页就会自动刷新

这也是推荐复用当前 `onWeeklyPlanChange()` 总线的原因。

## 11. 兼容策略

### 11.1 兼容旧 suggestion/proposal

历史消息里可能仍有旧 suggestion：

- 没有 `kind`
- 只有 `changes[]`

兼容规则：

- 没有 `kind` 时视为 `field_changes`
- 卡片照旧显示
- 旧 commit 逻辑照旧工作

### 11.2 兼容旧接口

旧接口：

- `/api/weekly-plan/adopt`

先继续保留，用于：

- 历史消息 fallback
- 低风险兼容

但新的单日训练计划卡不走这个接口，只走：

- `/api/tools/plan/commit`

### 11.3 兼容现有测试

已有覆盖“字段 patch proposal”链路的测试应继续通过。

新增测试应只补：

- 整日计划 proposal 生成
- commit 整日计划后写回
- 前端卡片渲染整日计划预览
- 前端采纳后训练计划页同步刷新

## 12. 测试策略

### 12.1 后端单元与接口测试

新增覆盖：

- `build_day_plan_replace_proposal()` 成功构造 proposal
- `validate_day_plan_payload()` 拒绝非法 day/type/exercise
- `POST /api/tools/plan/propose` 支持 `kind=day_plan_replace`
- `POST /api/tools/plan/commit` 成功写回目标日完整计划
- 重复 commit 被拒绝

### 12.2 前端单元测试

新增覆盖：

- `buildAdoptCardModel()` 支持 `day_plan_replace`
- `CoachTab` 在 commit 成功后保留本地顶层元数据并更新目标日
- `AdoptCard` 渲染动作预览列表

### 12.3 浏览器自动化

扩展现有 `tests/e2e/coach_browser_smoke.py`：

- mock 一个 `day_plan_replace` proposal
- 点击采纳
- 验证 commit 请求只发 `proposalId`
- 验证训练计划页出现新增/替换后的动作

## 13. 风险与注意事项

### 13.1 proposal payload 变大

整日计划 proposal 会比字段 patch 大很多。

但当前仍是本地 SQLite + 本地前后端 MVP，单日计划体量可接受。

### 13.2 动作归一化必须统一

AI 输出的动作结构不能直接原样写库，必须先走统一 normalize。

否则会出现：

- 训练计划页能显示一部分
- 但编辑器、负重显示、后续 AI 建议定位不到字段

### 13.3 不允许自动写回

即使模型已经拿到了整日计划结构，也必须继续走：

- proposal
- 用户确认
- commit

不能为了方便而放开自动写库。

## 14. 结论

V2.5 不需要从零发明“AI 采纳写回”能力，现有基础设施已经具备：

- suggestion/proposal 展示
- proposalId commit
- weeklyPlan 后端写回
- 前端全局状态更新

真正缺的是：

- proposal 能力从“字段 patch”升级到“单日训练计划卡”
- commit 能力从“只改已有字段”升级到“整日计划替换”
- 前端卡片从“变化清单”升级到“单日计划预览”

因此，本次推荐实现路径是：

1. 保留现有 `field_changes` proposal
2. 新增 `day_plan_replace` proposal
3. 复用现有 `/api/tools/plan/commit`
4. 在后端按 `proposal.kind` 分流写回
5. 在前端按 `proposal.kind` 渲染不同卡片

这条路径能最小化破坏现有链路，同时真正满足“确认一张单日训练计划卡后，agent 直接把对应动作和相关信息填入训练计划页”的目标。
