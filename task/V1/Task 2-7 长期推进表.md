## 执行原则

- 严格遵守 `plan -> code -> review`。
- 每个“子任务”都先由子 agent 编码，再由主 agent 复查与测试。
- 每个子任务完成后：
  - 主 agent 必须独立运行验证命令。
  - 验证通过后立即提交一次中文 git commit。
- 每个完整任务完成后：
  - 同步维护 `README.md`、`ARCHITECTURE.md`。
  - 补一轮任务级验证与收口。
  - 再提交一次中文 git commit。
- 不做空提交。任务级提交必须带有文档同步、结构收口或验收补充。
- 所有新增代码继续使用 UTF-8 与中文注释。
- 单文件超过 200 行时，优先在本任务内顺手拆分，避免把技术债拖到后面。

## 任务排序理由

- Task 2 和 Task 3 先做，是最小风险的数据输入层修正，能先稳住计划编辑体验。
- Task 4 是后续视觉与 AI 能力的地基，它同时牵动训练计划的数据结构、展示布局和 AI 可修改性，必须在主题统一与 AI 重构前完成。
- Task 5 放在 Task 4 后，是为了避免先美化旧布局、后面又重做一遍。
- Task 6 依赖 Task 5 后的稳定页面与上下文展示口径，同时会补充 prompt 注入的数据维度。
- Task 7 最后做，因为它依赖前面已经稳定下来的视觉语言、上下文结构与复杂指标输入。
