# V2.7 开发完成报告

## 1. 版本目标

V2.7 的目标不是新增 AI 功能，而是把当前已经可用的 AI 教练链路做一次“稳妥收口”：

- 明确前端和后端的唯一主入口
- 把 proposal 卡写回协议统一到 `commit / ignore`
- 把 provider runtime 固定为真实主路径
- 把历史命名、旧接口和兼容契约降级为兼容壳

## 2. 本轮完成内容

### 2.1 前端接口层收口

- `src/api/backendClient.js`
  - 新增 `commitCoachSuggestion()`
  - 规则为：优先走 `/api/tools/plan/commit`，只有历史 suggestion 缺少 `proposalId` 时才回退 `/api/weekly-plan/adopt`

- `src/tabs/CoachTab.jsx`
  - 采纳建议逻辑不再在页面层直接分叉 `commitPlanChange()` 和 `adoptWeeklyPlanChange()`
  - 页面层只调用 `backendClient.commitCoachSuggestion()`，进一步收窄页面职责

- `src/api/deepseek.js`
  - 保持兼容壳定位
  - 不再承担独立主流程，只继续转发到 `coachBackend.js`

### 2.2 后端聊天契约收口

- `backend/api/chat.py`
  - 新增 `NormalizedChatRequest`
  - 新增 `normalize_chat_request_payload()` 与 `normalize_stream_chat_request()`
  - `reply / stream / background` 三条入口统一先做请求归一
  - `userInput` 新契约和 `messages` 旧契约现在共享同一套内部解析结构

### 2.3 proposal 接口收口

- `backend/api/tools.py`
  - 明确 `/api/tools/plan/commit` 是当前 proposal 卡标准写回入口

- `backend/api/weekly_plan.py`
  - 为 `/api/weekly-plan/adopt` 增加 legacy compatibility shell 注释
  - 明确它只作为旧入口保留，不再作为 AI 教练主流程扩展点

### 2.4 provider runtime 与工具回环命名收口

- `backend/agent/chat_session.py`
  - 将 `_DeepSeekToolLoopProvider` 重命名为 `_ChatCompletionsToolLoopProvider`
  - 名称与职责现在一致：同时服务 legacy DeepSeek fallback 和 OpenAI-compatible `chat_completions` runtime

### 2.5 文档与口径收口

## 4. 产出结论

V2.7 完成后，项目已经形成更清晰的 AI 教练接口边界：

- 前端只有一套明确的 AI 聊天主入口
- proposal 卡只有一套明确的标准写回协议
- 后端聊天入口先归一再分发，不再三处并列分支
- provider runtime 主路径与 fallback 边界更清晰
- legacy 接口仍可兼容，但不再误导后续开发

## 5. 后续建议

- 若后续继续推进，可在更晚的版本里评估是否彻底删除前端 `deepseek.js`
- 在 legacy 迁移窗口结束后，可进一步压缩 `/api/weekly-plan/adopt` 与聊天 `messages` 旧契约的测试面
- 若未来新增 provider 类型，优先接入 provider runtime，不要再绕回 legacy client
