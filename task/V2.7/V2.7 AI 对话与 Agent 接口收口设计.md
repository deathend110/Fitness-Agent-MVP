# V2.7 AI 对话与 Agent 接口收口设计

## 1. 背景

当前 RepMind 的 AI 教练能力已经完成多轮迭代，形成了以下现状：

- 前端 AI 教练页已经稳定使用后端聊天代理，而不是浏览器直连模型。
- 后端已经形成 `chat -> agent context -> tool loop -> provider runtime` 的主链路。
- 多供应商模型配置、计划 proposal 卡、后台思考、文件摘要注入都已经接到这条链路上。

但与此同时，代码中仍保留了多批历史兼容接口、旧包装层和中间态抽象：

- 旧版 API 路由仍然存在，例如 `/api/weekly-plan/adopt`
- 前端仍保留带历史语义的 API 封装，例如 `src/api/deepseek.js`
- 后端仍存在“当前主链路已不使用，但仍作为 fallback / 兼容存在”的老客户端与旧契约
- 文档、测试和源码中同时存在“旧称呼 / 新实现 / 兼容别名”的混合表达

这会带来几个现实问题：

- 维护者很难判断“什么是现在真正的主入口”
- 新功能容易错误接到兼容壳上，而不是接到主链路
- 修 bug 时容易同时改动两套语义相近但职责不同的函数
- 测试和文档会被历史兼容语义持续拖累

因此需要做一次“稳妥收口”：

- 前端和内部实现统一到当前真实使用链路
- 后端对外旧接口短期保留兼容壳
- 兼容壳不再继续承载主流程演进
- 文档、测试、命名和模块职责向统一后的主链路收拢

## 2. 目标

本轮目标不是做功能新增，而是做架构收口与接口去重。

目标如下：

1. 明确并固化当前 AI 对话与 agent 的唯一主链路
2. 去除当前主链路之外的重复包装函数、重复调用入口和无实际价值的历史中间层
3. 把旧接口降级为最薄兼容壳，而不是继续参与业务分支
4. 让前端、后端、工具调用、provider runtime 和文档口径保持一致
5. 保证当前功能不回退：
   - 普通聊天
   - 流式聊天
   - 后台思考
   - proposal 计划卡生成
   - commit / ignore
   - 多模型 provider 选路
   - DeepSeek / OpenAI-compatible / Gemini 运行时

## 3. 非目标

本轮明确不做以下事项：

- 不重写 AI 教练页面 UI
- 不重做聊天消息数据结构
- 不删除所有 legacy 路由
- 不彻底删除 `DeepSeekClient`
- 不调整数据库表结构
- 不重构与 AI 教练无关的训练计划页、档案页、今日日志页

## 4. 当前真实主链路

### 4.1 前端 AI 教练主链路

当前前端真正使用的 AI 教练主链路为：

`src/tabs/CoachTab.jsx`

通过以下函数与类协作：

- `src/utils/coachChat.js`
  - `requestCoachReply()`
  - `requestCoachReplyStream()`
  - `startBackgroundCoachReply()`
  - `getBackgroundCoachTask()`
- `src/api/coachBackend.js`
  - `/chat/reply`
  - `/chat/stream`
  - `/chat/{session_id}/background`
  - `/chat/background/{task_id}`
- `src/api/backendClient.js`
  - 会话、草稿、文件、模型配置、计划卡 commit / ignore 等非聊天流式接口

当前推荐口径：

- AI 对话请求只从 `coachChat.js` 进入
- AI 教练相关普通后端 CRUD 只从 `backendClient.js` 进入
- `CoachTab.jsx` 不直接拼装 fetch URL，不直接持有独立 API 逻辑

### 4.2 后端 AI 教练主链路

当前后端真正使用的 AI 主链路为：

- `backend/api/chat.py`
  - 统一聊天入口
  - 聊天会话持久化
  - 流式代理
  - 非流式回退
  - proposal 纠偏与消息落库
- `backend/agent/chat_session.py`
  - 统一 agent 编排入口
  - 上下文读取与装配
  - provider runtime 绑定
  - tool loop 接线
- `backend/agent/tool_loop.py`
  - 协议无关的工具回环
- `backend/agent/tool_calling.py`
  - 工具注册和参数校验
- `backend/agent/background_worker.py`
  - 离页后台思考任务

### 4.3 运行时主链路

当前模型调用主链路已经是：

- `backend/model_config/runtime.py`
- `backend/agent/chat_session.py::build_provider_bound_client()`
- `backend/agent/chat_session.py::run_tool_calling_chat()`

provider runtime 的统一口径是：

- DeepSeek：默认走 OpenAI-compatible runtime
- OpenAI-compatible：走 `OpenAICompatibleRuntimeClient`
- Gemini-native：走 `GeminiNativeClient`

当前 `DeepSeekClient` 的正确定位已经不是主链路，而是：

- fallback
- 兼容旧测试替身
- provider runtime 缺失时的兜底

### 4.4 计划卡主链路

当前结构化计划卡的真实主链路是：

- 生成：tool loop 中的 proposal 工具
- 写回：`POST /api/tools/plan/commit`
- 忽略：`POST /api/tools/plan/ignore`
- 前端展示：`buildAdoptCardModel()`
- 前端触发：`CoachTab -> backendClient.commitPlanChange()/ignorePlanChange()`

这条链路已经覆盖：

- 字段 patch proposal
- 单日训练计划替换 proposal
- 写回训练计划
- suggestion 状态同步回写 `chat_message`

## 5. 当前冗余与分叉点

### 5.1 前端冗余入口

#### A. `src/api/deepseek.js`

问题：

- 命名仍然是旧时代的 DeepSeek 直连语义
- 实际项目当前已经统一走后端代理
- 与 `coachChat.js` / `coachBackend.js` 存在语义重叠

结论：

- 不应再作为主入口
- 本轮应评估是否保留为薄兼容壳，或由内部转发到新接口
- 文档与主代码不再引用它

#### B. `CoachTab.jsx` 中局部重复逻辑

问题：

- 既使用 `coachChat.js`，又大量直接依赖 `backendClient.js`
- 会话、草稿、背景任务、计划卡处理逻辑全部堆在页面组件里
- 逻辑虽然可运行，但边界过宽

结论：

- 本轮不要求大拆 UI 结构
- 但要清理重复调用和错误复用途径
- 页面内只保留协调逻辑，不保留历史兼容思维分支

### 5.2 后端冗余路由

#### A. `/api/weekly-plan/adopt`

问题：

- 这是旧版“直接按 day + changes 提交写回”的接口
- 当前 AI proposal 卡主流程已经统一到 `/api/tools/plan/commit`
- 前端 AI 教练 proposal 主流程已不应依赖它

结论：

- 保留为 legacy 兼容壳
- 不再作为 AI 教练主流程文档入口
- 新代码、主流程、README/ARCHITECTURE 不再推荐使用它

#### B. `POST /api/chat/reply` 与 `GET /api/chat/stream` 的双契约

当前同时支持：

- 旧契约：`messages`
- 新契约：`userInput + sessionId + model + thinking + fileIds`

问题：

- 新前端实际上已经基于新契约在运行
- 旧契约继续存在会让 `chat.py` 长期承担双分支校验与解析负担

结论：

- 对外暂不删除旧契约
- 但内部要统一把两类输入尽快归一到一份规范化结构
- 文档、测试和主要调用路径只强调新契约

### 5.3 后端运行时冗余

#### A. `DeepSeekClient` 与 `OpenAICompatibleRuntimeClient` 双中心问题

问题：

- 源码和测试中历史上长期把 `DeepSeekClient` 视为主路径
- 但现在真实主路径已转到 OpenAI-compatible runtime
- 如果文档和命名不收口，后续维护仍会误接到 `DeepSeekClient`

结论：

- 保留 `DeepSeekClient`，但降格成明确 fallback
- 所有主流程描述都要改为 provider runtime 优先
- 新功能不得再以 `DeepSeekClient` 为默认主入口

#### B. provider wrapper 的命名和职责混合

当前 `_DeepSeekToolLoopProvider` 实际承载的是：

- legacy DeepSeekClient
- OpenAI-compatible chat_completions runtime

问题：

- 名称容易误导
- 会让人以为它只服务 DeepSeek

结论：

- 本轮应评估重命名为更中性的 chat-completions provider wrapper
- 避免“实现已泛化，命名仍是 DeepSeek 专属”的错觉

### 5.4 文档与测试冗余

问题：

- README、ARCHITECTURE、backend/README 中同时存在历史兼容入口和当前主流程表述
- tests 中存在大量“源码字符串断言型测试”，用于锁 UI 契约和调用契约
- 其中一部分仍引用 legacy 名称或旧 API

结论：

- 文档要收口到“主链路 + 兼容壳说明”
- 测试要区分：
  - 主流程测试：继续增强
  - 兼容壳测试：保留最小必要覆盖
  - 历史命名遗留测试：应删或改名

## 6. 本轮统一后的目标结构

### 6.1 前端接口层统一

#### 聊天

- 保留：
  - `src/utils/coachChat.js`
  - `src/api/coachBackend.js`
- 目标：
  - `coachChat.js` 负责 AI 教练业务请求语义
  - `coachBackend.js` 负责聊天相关 HTTP 封装

#### 普通 CRUD / 会话 / 草稿 / 文件 / 工具

- 保留：
  - `src/api/backendClient.js`
- 目标：
  - 所有非聊天流式接口统一归 `backendClient.js`

#### 历史兼容

- `src/api/deepseek.js`
  - 若仍有引用，则改为纯转发壳
  - 若无生产引用，只保留最小兼容或删除

### 6.2 后端聊天统一

- 主入口：
  - `backend/api/chat.py`
- 主编排：
  - `backend/agent/chat_session.py`
- 主工具回环：
  - `run_tool_calling_chat()`
- 主 provider 选路：
  - `build_provider_bound_client()`

要求：

- 新旧请求体都先归一成统一输入模型
- 主流程不再出现多套并列分支
- provider-aware 错误提示统一收口

### 6.3 计划卡统一

- 统一 proposal 语义
- 统一 commit / ignore
- `/api/weekly-plan/adopt` 只保留 legacy 兼容

要求：

- `CoachTab` 采纳 proposal 后不再走旧 adopt 语义分支
- 对于无 `proposalId` 的历史 suggestion，明确是否继续兼容
- 文档只把 `commit / ignore` 视为主接口

### 6.4 provider runtime 统一

- 主路径：
  - OpenAI-compatible runtime
  - Gemini-native runtime
- fallback：
  - `DeepSeekClient`

要求：

- 所有聊天、流式、后台任务都先走 provider runtime
- `DeepSeekClient` 只在 runtime 不可用时兜底
- tool loop provider wrapper 命名与职责保持一致

## 7. 删除与保留原则

### 7.1 可以删除

满足以下条件的函数 / 接口可删除：

1. 当前生产链路无引用
2. 当前测试仅因历史名称保留、无真实业务价值
3. 可以由统一后的主接口完全覆盖
4. 删除不会导致旧数据结构无法读取

### 7.2 必须保留兼容壳

满足以下条件的函数 / 接口本轮只降级不删除：

1. 仍有测试或脚本依赖
2. 仍承担旧数据或旧请求格式兼容
3. 删除会导致当前联调路径直接中断
4. 后续仍需一个迁移窗口

包括但不限于：

- `/api/weekly-plan/adopt`
- `/api/chat/reply` / `/api/chat/stream` 中的旧 `messages` 契约
- `DeepSeekClient`

### 7.3 不应继续扩展

以下对象即使保留，也不允许再继续承载新功能：

- legacy adopt 路由
- legacy plain messages 聊天契约
- 历史命名 API 包装层
- 仅为旧实现保留的中间解析函数

## 8. 风险

### 8.1 兼容壳删早了

风险：

- 旧测试大面积失败
- 浏览器自动化脚本或本地验证脚本失效

应对：

- 先把主流程统一
- 再把兼容壳压薄
- 最后评估是否删除

### 8.2 文档先改、代码没收口

风险：

- README/ARCHITECTURE 与实际实现不一致

应对：

- 每一轮收口任务都同步更新文档
- 文档明确标出“主链路”和“兼容壳”

### 8.3 测试覆盖误导

风险：

- 源码字符串测试通过，但真实链路已漂移

应对：

- 保留最小字符串契约测试
- 增强真正打 API / 真正走 client 的集成测试

## 9. 实施原则

本轮实施按以下顺序推进：

1. 前端接口层收口
2. 后端聊天入口与 proposal 提交流口收口
3. provider runtime 与 tool loop 命名 / 入口收口
4. 文档与测试清理

每一步都必须满足：

- 小步修改
- 先补失败测试再改实现
- 每步通过回归后立即提交中文 commit

## 10. 预期结果

完成后应达到以下状态：

- 新开发者能明确知道 AI 教练只有一套主调用链
- proposal 卡只有一套标准写回协议
- provider runtime 只有一套标准选路入口
- legacy 接口仍能短期兼容，但不会再误导主流程
- README / ARCHITECTURE / backend README 对同一件事只有一套主说法

---

文档结论：

本轮应采用“稳妥收口”策略，而不是激进删除。核心原则是：

- 主链路唯一
- 兼容壳变薄
- 历史命名降权
- 文档和测试跟着主链路走

在这个前提下，再进入实施计划与分任务编码。
