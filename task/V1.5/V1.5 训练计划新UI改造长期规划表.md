# V1.5 训练计划新 UI 改造长期规划表

## 范围说明

- 本规划覆盖“基于 `训练计划页面效果图.html` 重构当前应用整体 UI 风格”的工作。
- 本轮优先级以训练计划页、应用壳层，以及其余 tab 的同风格迁移为主。
- 原 `V1 Task 7 - AI 对话框能力完善` 不再作为当前阶段工作，已后移到 `V2`，本规划不处理 AI 对话页重构本体。
- 当前目标不是新增完整新业务，而是在尽量复用现有训练计划、档案、今日日志、AI 数据链路的前提下，把 UI 结构升级到新设计稿对应的产品形态。
- 当前阶段不仅训练计划页要贴近新设计，其对应的“我的档案”“今日日志”“AI 教练”也要同步迁移到同一套新壳层与页面布局语言。

## 为什么这是大改动

这次改动应视为 **大改动**，不适合塞进 `task/V1` 的单个后续任务，原因如下：

1. `App.jsx` 级别的应用壳层会从“顶部标题 + Tab 按钮”切换为“左侧侧栏 + 主内容区”的新布局。
2. 训练计划页不只是样式变化，还新增了周区间头部、周编号、图例、快捷操作区等页面结构。
3. 周视图列模板发生明显变化：训练日宽列、休息日窄列、动作卡片密度更高、信息组织方式不同。
4. 一部分新 UI 会牵出轻量逻辑变化，例如：
   - 计划设置入口占位
   - 训练计划头部周元信息展示
5. 现有训练计划编辑交互是“展开式编辑卡片”，新设计稿更偏“看板式展示 + 局部入口编辑”，需要重排组件边界。
6. 用户已明确要求其余 tab 也要跟随新 UI 同步变化，这意味着 `ProfileTab / TodayTab / CoachTab` 的页面布局都要一起重构，而不是只改一个 `PlanTab`。
7. 用户已明确要求在全屏 16:9 桌面视口下禁止左右滑动，这会直接影响训练计划页网格比例、列内容密度和应用整体宽度策略。

因此，本次工作更适合作为 `V1.5` 独立规划执行。

## 目标

把当前项目的训练计划相关体验升级为更接近效果稿的产品形态，满足以下目标：

- 应用壳层接近效果稿：左侧导航、底部状态区、主内容区分栏明确。
- 训练计划页成为更强的“周训练看板”，一眼看出本周结构。
- “我的档案”“今日日志”“AI 教练”在布局语言、间距、层级、卡片结构上与新壳层统一。
- 在正常 16:9 全屏桌面视口下，训练计划页禁止左右滑动，只允许页面纵向滚动。
- 在正常 16:9 全屏桌面视口下，页面应完整展示当前层级下的关键信息，优先通过信息密度、排版层级、分区组织解决，而不是依赖裁切、遮挡或把核心信息折叠到不可见位置。
- 保留并兼容当前 MVP 已有的训练计划数据能力，不为追求样式而破坏现有闭环。
- 为后续 `V2` 的 AI 页、DeepSeek API 优化、Agent 优化留出更稳定的 UI 与数据边界。

## 当前已知约束

- 继续遵守 `plan -> code -> review`。
- 所有代码保持 UTF-8 与中文注释。
- 单文件超过 200 行时必须评估拆分。
- 每个实现子任务完成后都要独立测试、独立提交中文 git commit。
- 当前工作区存在用户自己的未提交改动，执行时不能回退无关文件。
- 在 16:9 全屏桌面视口下禁止依赖横向滚动承载训练计划主布局。
- 若因更窄屏幕需要降级，应优先通过更紧凑布局、断行和信息压缩处理；横向滚动只能作为非目标视口的兜底，而不是主方案。
- 不能为了消除横向滚动而简单减少信息量；关键业务信息必须仍然在页面主视图中完整可见。

1. 先改应用壳层，再改训练计划页骨架。
2. 优先解决“16:9 全屏无横向滚动”的布局硬约束。
3. 再做周看板内部的训练日 / 休息日表现与编辑交互。
4. 训练计划页稳定后，再把档案、今日日志、AI 教练三页同步迁移到新风格。
5. 收口阶段统一补 README / ARCHITECTURE / 验收记录。

## 风险提醒

- `App.jsx` 会被重构，影响所有 tab 的承载方式，必须先做边界设计，避免一次把四个页面都改乱。
- `PlanTab.jsx`、`PlanDayCard.jsx` 以及若干计划组件很可能再次触碰 200 行阈值，需要提前拆解责任。
- 效果稿中存在“计划设置”这类未来功能入口，但当前阶段只保留占位，不实现复杂周期计划和经典计划库本体。
- “全屏禁止左右滑动”会反向约束训练计划卡片的密度、文字截断和列宽算法，不能按旧方案把横向滚动当正常体验。
- “完整展示所有信息”意味着需要更强的信息层级设计，不能靠大面积折叠、过度省略文案或把关键字段藏进二级入口来换布局空间。
- 其余三个 tab 若只换颜色不换布局，会让整个应用出现明显割裂感，因此必须作为 `V1.5` 范围的一部分统一处理。
- 不能为了贴稿而一次性引入新的复杂后端概念或大规模状态管理。
