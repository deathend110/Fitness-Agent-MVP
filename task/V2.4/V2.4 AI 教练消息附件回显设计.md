# V2.4 AI 教练消息附件回显设计

> 日期：2026-06-01
> 范围：AI 教练上传文件在“用户消息气泡”中的持久化回显
> 上游背景：[V2.3 开发完成总结](<../V2.3/V2.3 开发完成总结.md>)

## 1. 背景与问题

当前 Phase 4 已经支持：

- 上传文件到 `UploadedFile`
- 在输入区通过 `FileAttachmentTray` 显示待发送附件
- 聊天请求携带 `fileIds`
- 后端将文件摘要注入 Agent 上下文

但当前链路里，附件只存在于：

- 前端 `attachedFiles` 草稿态
- 后端 `CoachDraft.attached_file_ids`
- 后端 `UploadedFile`

真正持久化的 `ChatMessage` 没有附件字段。结果是：

- 用户发送成功后，输入区附件会被清空
- 历史消息列表不知道“这条用户消息到底带了哪些文件”
- 刷新页面、切换历史会话、重新进入会话后，消息气泡无法回显附件

这和常见大模型网页端的行为不一致。主流产品通常会把附件作为“消息的一部分”展示，而不是只作为发送前的输入控件状态。

## 2. 目标

本次 V2.4 目标是让 AI 教练页的附件表现与常见大模型网页端一致：

- 用户发送消息时，将当次附件绑定到该条用户消息
- 用户消息气泡在正文上方显示附件卡片
- 刷新页面、切换历史会话、重新进入会话后，附件卡片仍可回显
- AI 回复气泡不重复显示这些附件

本次确认后的展示密度为“标准版”：

- 文件图标
- 文件名
- 类型/大小

## 3. 非目标

本次不做：

- 文件在线预览弹窗
- 点击附件重新下载或打开原文件
- AI 回复气泡的附件回显
- 图片缩略图、DOCX 首图、Excel 表格预览
- 跨消息复用同一附件卡片组件做复杂交互
- 向量检索、OCR、文件引用高亮

## 4. 推荐方案

推荐采用“消息级附件快照”方案：

- 为 `ChatMessage` 增加 `attachments` JSON 字段
- 每条用户消息在写入时保存一份最小附件快照
- 消息读取接口直接返回这份快照
- 前端消息列表直接消费 `message.attachments`

每个附件快照保存以下字段：

- `fileId`
- `originalName`
- `mimeType`
- `extension`
- `sizeBytes`

推荐该方案的原因：

- 最接近主流聊天产品的交互心智模型
- 刷新和历史恢复稳定，不依赖输入区状态
- 即使后续 `UploadedFile` 被删除，消息仍能保留当时的展示信息
- 前端恢复消息时不需要额外再查一次文件详情才能渲染

不采用“只存 fileIds、读取消息时再查 UploadedFile”的原因：

- 会增加历史恢复时的额外请求
- 文件被删除或元数据变化后，历史消息显示会漂移
- 更像“引用当前文件表”，不像“保存当时发送出去的消息快照”

## 5. 数据结构设计

### 5.1 ChatMessage 增强

`backend/db/models.py` 中的 `ChatMessage` 新增：

```json
"attachments": [
  {
    "fileId": 12,
    "originalName": "减脂容量型计划.xlsx",
    "mimeType": "application/vnd.openxmlformats-officedocument.spreadsheetml.sheet",
    "extension": ".xlsx",
    "sizeBytes": 10321
  }
]
```

设计约束：

- 只保存最小展示快照，不嵌入文件全文、不嵌入解析摘要
- `attachments` 默认为空数组
- 旧消息没有该字段时按空数组兼容

### 5.2 Schema 增强

`ChatMessageCreateSchema` 与 `ChatMessageSchema` 同步增加 `attachments` 字段。

后端内部约束：

- `assistant/system` 消息允许为空数组
- `user` 消息可带附件数组
- 后端不信任前端任意结构，需对字段做基本裁剪和白名单保留

## 6. 前后端数据流

### 6.1 发送链路

发送前：

- `CoachTab` 维护 `attachedFiles`
- 每个元素当前已经包含上传完成后的 metadata

点击发送时：

- `CoachTab` 先从 `attachedFiles` 生成 `messageAttachmentSnapshots`
- 立即把这份快照挂到本地 `userMessage.attachments`
- 聊天请求仍只提交现有 `fileIds`

后端持久化时：

- 后端根据本轮 `fileIds` 从 `UploadedFile` 生成一份最小附件快照
- `persist_successful_chat_turn()` 在写入 user message 时保存 `attachments`
- assistant message 保持 `attachments=[]`

### 6.2 历史恢复链路

恢复消息时：

- `GET /api/chat/sessions/{id}/messages` 返回 `attachments`
- `CoachTab.normalizeChatMessages()` 保留 `attachments`
- `MessageList -> MessageBubble` 渲染用户附件卡片

因此：

- 刷新页面后可回显
- 切换会话后可回显
- 旧消息无附件时不受影响

### 6.3 后台任务链路

后台思考只补 assistant 结果，不需要为 assistant 添加附件卡片。

但为保证链路统一：

- 前台正常发送时的 user message 必须已经带附件快照
- 后台任务只负责补 assistant，不重写 user message

这样可以避免“离页后结果补回，但用户消息附件消失”的不一致问题。

## 7. UI 设计

### 7.1 展示位置

用户消息气泡改成两段式：

1. 附件卡片区
2. 文本气泡

附件卡片位于文本上方，与常见大模型网页端一致。

### 7.2 卡片信息密度

本次采用标准版：

- 左侧文件类型图标
- 第一行文件名
- 第二行类型标签 + 大小

示例：

- `减脂容量型计划.xlsx`
- `XLSX 10.18KB`

### 7.3 视觉要求

- 保持当前浅色工作台语言
- 卡片宽度小于或等于消息气泡宽度
- 多附件时纵向堆叠
- 不引入厚重操作按钮，不做预览/下载入口

### 7.4 输入区与消息区的职责划分

输入区的 `FileAttachmentTray` 继续表示“待发送附件”。

发送成功后：

- 输入区附件清空
- 历史消息卡片保留

这两个区域分别表达：

- 发送前：我准备发送什么
- 发送后：这条消息实际带了什么

## 8. 边界情况

### 8.1 删除原文件

如果后续用户删除 `UploadedFile`：

- 历史消息仍显示附件快照
- 但不保证能重新打开原文件

这是有意设计，优先保证聊天记录的一致性。

### 8.2 旧消息兼容

旧 `chat_message` 行没有 `attachments`：

- API 返回空数组
- 前端不报错
- UI 不显示附件区

### 8.3 多附件

一条用户消息可以带多个附件。

MVP 阶段：

- 顺序按发送时选择顺序保留
- 采用纵向堆叠
- 不做“+N”折叠

### 8.4 上传失败附件

上传失败的文件不会进入 `attachedFiles` 的成功发送快照，不应进入消息附件区。

只有真正已经上传成功并参与本次请求的文件，才应附着到用户消息。

## 9. 测试策略

至少覆盖以下验证：

### 9.1 成功路径

- 上传文件
- 输入问题并发送
- 用户消息气泡显示附件卡片
- 刷新页面后仍显示
- 切换会话后重新回来仍显示

### 9.2 边界情况

- 同一条消息附带多个文件
- 没有附件的普通消息
- 旧历史消息无 `attachments`

### 9.3 失败场景

- 上传失败文件不进入消息附件卡片
- 文件删除后历史消息仍能显示快照
- 后台任务补 assistant 时不污染用户附件显示

## 10. 影响范围

本次改动会影响：

- 后端 `ChatMessage` 模型与迁移
- 聊天消息 schema
- 聊天持久化逻辑
- 前端 `CoachTab` 本地消息结构
- 用户消息气泡渲染
- 聊天相关前后端测试

不会影响：

- 周计划采纳逻辑
- 文件上传解析逻辑
- 模型选择、thinking、Markdown 渲染
- 今日日志与训练计划页

## 11. 结论

V2.4 应以“消息级附件快照”实现 AI 教练消息附件回显。

这是当前仓库中最小但完整、也最符合真实产品体验的实现路径。它不需要引入额外复杂系统，却能把“附件只是输入态”升级为“附件属于消息本身”，从而补齐用户对聊天记录可信度与可追溯性的预期。
