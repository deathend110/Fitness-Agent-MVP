# V2.3 Python 后端基建长期规划表

> 上游设计：[docs/RepMind Python 重构设计.md](<../../docs/RepMind Python 重构设计.md>)
> 本轮范围：设计文档里的 **Phase 1 + Phase 2**（后端骨架/数据迁移 + Python Agent/密钥后移）
> 基准版本：V2.1（纯前端 + localStorage），目标版本：V2.3（前后端分离，本地 Python 后端）

## 范围说明

- 本轮只做设计文档的 **Phase 1 + Phase 2**，一次性解除「存储 · 密钥 · 后台」三大硬约束：
  - **Phase 1**：新建 `backend/`（FastAPI + SQLite），把 `profile / weeklyPlan / dailyLog` 迁到后端 CRUD，前端从直读 localStorage 改为调后端；提供 localStorage→DB 一次性导入与后端不可用降级。
  - **Phase 2**：DeepSeek 调用后移（API Key 只存后端）、SSE 流式回复、离页后台思考、`chat` 全量落库（去掉 20 条裁剪）、计划采纳校验后端化。
- **本轮明确不做**（留后续版本 / Phase 3–5）：
  - 文件上传解析、模型选择下拉、前端 MD 渲染、未发送草稿后端持久化（→ Phase 4 / V2.5 #0）
  - system prompt 与复杂指标迁移、上下文压缩 / token 优化、Function Calling 完整链路（→ Phase 3）
  - 真实多会话 UI、跨对话知识库、周期计划引擎 / 预制计划库（→ Phase 5 / V3）

## 现状结论（V2.1，已读码核实）

- [src/App.jsx](<../../src/App.jsx>) 持有 `profile / weeklyPlan / dailyLog / chatHistory` 四份顶层 state，启动 `loadInitialState` 读 localStorage，四个 `useEffect` 调 `saveStorage` 写回——这是「前端切源」的唯一改造点。
- [src/utils/storage.js](<../../src/utils/storage.js>) 仅封装 localStorage 读写；键名在 [src/utils/defaultData.js](<../../src/utils/defaultData.js>) 的 `storageKeys`（`fitloop_profile/weeklyPlan/dailyLog/chatHistory/storageVersion`）。
- [src/utils/dataTransfer.js](<../../src/utils/dataTransfer.js>) 的 `buildBackupPayload` 已产出 `{profile, weeklyPlan, dailyLog, chatHistory}` 形状，可直接作为 localStorage→DB 导入载体。
- [src/api/deepseek.js](<../../src/api/deepseek.js>) 浏览器直连 DeepSeek，`VITE_DEEPSEEK_API_KEY` 打包进前端、`DEFAULT_MODEL` 硬编码——密钥后移与 SSE 代理的改造点。
- [src/utils/coachChat.js](<../../src/utils/coachChat.js>) 组装 `[system, ...chatHistory, user]`，`buildPromptImpl / requestImpl / streamImpl` 均可注入——便于退化为后端薄包装。
- [src/utils/chatHistory.js](<../../src/utils/chatHistory.js>) `CHAT_HISTORY_LIMIT=20` 硬裁剪；[src/utils/adoptPlan.js](<../../src/utils/adoptPlan.js>) 的 `adoptPlanChange` 是计划写回校验的唯一来源，将被搬到后端复用。

## 关键架构决策（落地前必须对齐）

1. **不搬 `frontend/` 目录**：前端继续留在仓库根 `src/`，`backend/` 作为兄弟目录新增，降低本轮 churn；设计文档第六节的 `frontend/` 重组留到后续版本，可选。
2. **数据契约零改写**：`weekly_plan_day.day_key` 用全名星期键 `Monday..Sunday`（与现有代码一致，非 `Mon..Sun`）；`exercises` 整存 **JSON 列**，原样保留 `template + instance + 扁平兼容字段`；`profile / dailyLog` 形状不变；统一用 pydantic schema 固化契约。
3. **Phase 2 后端是「瘦聊天代理」**：system prompt 仍由前端构建（复用现有 `prompt.js` + 指标单一口径 `buildDailyMetricsSummary`），前端把 `messages[]` 发给后端，后端只负责**藏密钥 + SSE 流式 + 消息落库 + 后台任务**。prompt / 指标 / 上下文管理的迁移属 Phase 3，不在 V2.3。
4. **测试策略**：前端继续用 Node 内置 `node:test`（不引 DOM）；后端新增 `pytest + pytest-asyncio`。每个子任务验证 = 定向测试（pytest 或 node --test）+ `npm run build` + 必要的手动联调。
5. **密钥纪律**：`DEEPSEEK_API_KEY` 只存 `backend/.env`，前端永不接触；缺失时后端接口返回明确错误、前端给友好降级。`backend/.env` 与 `backend/data/`（repmind.db + 未来 uploads）进 `.gitignore`。

## 协作与验收约定（对齐 AGENTS.md）

- 编码默认走 `superpowers:subagent-driven-development`，主 agent 负责规划/拆分/派发与**两轮审查**（先核对计划，再查代码质量 + 验证 + 文档同步）。
- **每个子任务验证通过后立即 `git commit`，commit message 用中文**，准确描述本次改动核心（见各实施计划的 Commit 行）。
- 文档同步是硬性要求：触碰架构的任务需同步 [README.md](<../../README.md>) 与 [ARCHITECTURE.md](<../../ARCHITECTURE.md>)，并新增 `backend/README.md`；本轮在两个 Phase 的收口任务（Task 6 / Task 12）集中收口文档。
- 单文件超 200 行必须评估拆分；后端按 `db / api / agent` 分层，避免巨型文件。
- 每个 Task 的验收对齐「1 条核心成功路径 / 2–3 个边界 / 1 个失败场景」。

## 风险提醒

- **数据迁移丢失**：导入接口必须幂等可重试；迁移前自动 JSON 备份（复用 `buildBackupPayload`）；导入后保留 localStorage 原值直到用户确认。
- **前后端契约漂移**：pydantic schema 固化 + `exercises` JSON 往返测试；保持现有数据形状不变。
- **本地双进程启动复杂**：根目录加 `npm run dev:all`（`concurrently` 同起 Vite + uvicorn），README 写清。
- **SSE 断连**：前端流式失败回退后端非流式接口；离页用后台任务兜底，回页拉取最新结果。
- **范围蔓延**：不要把 V2.3 扩成模型选择 / 文件上传 / 知识库 / 周期计划——这些是 Phase 3–5 / V2.5 / V3。
