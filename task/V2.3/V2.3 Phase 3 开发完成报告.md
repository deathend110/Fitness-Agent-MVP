## 目标

Phase 3 的目标是把 Phase 2 的“后端聊天代理”升级为真正的本地健身 Agent 编排层。前端只提交用户输入和模型配置，后端负责读取 SQLite 状态、管理上下文窗口、压缩长对话、记录 token/cache 使用、检索长期 memory，并通过 DeepSeek Tool Calls 读取训练相关状态和生成计划修改 proposal。

## 架构变化

Phase 3 后的核心链路：

```text
CoachTab
  -> src/api/coachBackend.js
  -> /api/chat/stream 或 /api/chat/reply
  -> backend.agent.chat_session.build_agent_request()
  -> PromptAssembler / SummaryCompressor / MemoryRetriever
  -> run_tool_calling_chat()
  -> DeepSeek
  -> proposal / suggestion
  -> 用户确认
  -> /api/tools/plan/commit
  -> weekly_plan_day 写回
```

关键边界：

- `memory_item` 保存用户长期事实，`knowledge_item` 保存外部资料或上传文件知识，两者分离。
- `chat_session_summary` 只用于压缩上下文，不替代完整 `chat_message`。
- `usage_record` 只记录 DeepSeek 返回的真实 usage，不伪造流式缺失字段。
- `tool_call_log` 记录工具参数、结果摘要、状态和错误，便于审计。
- 模型不能直接写计划，只能生成 proposal；写库必须来自用户确认路径。

## 已知限制

- 前端真实多会话 UI 尚未完成，当前 AI 教练页面展示仍主要依赖本地 `fitloop_chatHistory`。
- `read_uploaded_file_summary` 仍是 Phase 4 文件上传解析的占位工具。
- memory 检索为规则/关键词 MVP，未接向量检索。
- `calculate_metrics` 先实现轻量指标，完整 Python 指标服务后续迁移。
- proposal 存储为进程内 MVP，服务重启会导致未提交 proposal 失效。
