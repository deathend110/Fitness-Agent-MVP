# V2.3 Phase 3 上下文管理与工具调用实施计划

> **For agentic workers:** REQUIRED SUB-SKILL：使用 `superpowers:subagent-driven-development`（推荐）或 `superpowers:executing-plans` 按任务执行。步骤用 checkbox 语法追踪。
> 上游：[V2.3 Python 后端基建长期规划表](<./V2.3 Python 后端基建长期规划表.md>) / [docs/RepMind Python 重构设计.md](<../../docs/RepMind Python 重构设计.md>)
> 前置：Phase 1 + Phase 2 必须全绿。当前已具备 FastAPI + SQLite、主数据 CRUD、chat 全量存储、DeepSeek 后端代理、SSE、离页后台思考、计划采纳后端校验。

**Goal:** 把 Phase 2 的「瘦聊天代理」升级为真正的健身 Agent 编排层：后端统一构建 prompt、管理上下文窗口、压缩长对话、记录 token/cache 使用、沉淀长期 memory，并接入 DeepSeek Tool Calls。模型可以读取档案/计划/日志/指标/记忆并提出计划修改，但真正写回训练计划仍必须经过用户确认。

**Architecture:** Phase 3 后端是「Agent Orchestrator」。前端不再构建 system prompt，也不再把整段 `messages[]` 原样传给后端；前端只提交 `sessionId + userInput + model/thinking 配置 + 可选附件引用`。后端从 SQLite 读取 profile / weeklyPlan / dailyLog / chatHistory / session summary / memory / knowledge，按稳定前缀、领域状态、检索记忆、最近对话和当前输入组装 DeepSeek `messages`。DeepSeek Context Caching 依赖稳定前缀，因此 system prompt 与工具 schema 要保持稳定，动态数据后置。Tool Calls 采用「只读工具 + propose 写入建议 + 用户确认 commit」三段式，避免模型静默改坏训练计划。

**Tech Stack:** 续 Phase 1/2（FastAPI / SQLAlchemy async / pytest / httpx / SSE / asyncio）。新增后端模块：`context_manager.py`、`memory.py`、`usage_ledger.py`、`tool_calling.py`、`prompt_templates.py`。Token 计数 MVP 先使用保守估算器（便于测试稳定），后续可替换为 DeepSeek 官方 tokenizer 或兼容 tokenizer；所有真实成本统计以后端记录 DeepSeek `usage` 为准。

**External References:**
- Claude Code 分析仓库：`https://github.com/liuup/claude-code-analysis`。借鉴上下文治理思想：分层记忆、压缩摘要、工具结果脱水、压缩失败熔断和状态回注；不照搬 coding agent 的 shell/file edit 能力。
- DeepSeek API 文档：`https://api-docs.deepseek.com/`。实现必须遵守 DeepSeek `/chat/completions`、Thinking Mode、Tool Calls、Context Caching、usage cache hit/miss 字段等接口语义。

## Phase 3 总体验收清单

- [ ] **核心成功路径**：用户在 AI 教练页提问“根据我这周深蹲表现和今天疲劳，帮我调整明天训练”，后端自动读取 profile/weeklyPlan/dailyLog/memory，必要时调用工具，AI 返回解释与计划修改建议卡；用户点击采纳后计划写回 SQLite，刷新仍保留。
- [ ] **长对话边界**：构造 80+ 轮历史后发送新问题，后端触发摘要或复用已有 summary，不超 token 预算；回答仍保留安全限制、当前计划和用户长期偏好。
- [ ] **memory 边界**：用户明确说“我膝盖深蹲到底部会疼”，该信息进入 safety memory 并优先注入；用户只说“今天很累”不会晋升长期 memory。
- [ ] **工具边界**：模型请求不存在动作名或非法 RPE，工具返回错误，计划不变；前端显示可理解提示。
- [ ] **缓存与成本观测**：`/context/debug` 能看到本轮估算 token、真实 usage、prompt cache hit/miss；稳定前缀不被动态时间戳破坏。
- [ ] **失败场景**：DeepSeek 超时、工具循环超限、摘要压缩失败、proposal 重复提交都不会写脏数据，也不会保存半截 assistant。
