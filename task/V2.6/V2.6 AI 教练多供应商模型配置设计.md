# V2.6 AI 教练多供应商模型配置设计

> 日期：2026-06-01
> 范围：在 AI 教练页面内增加“模型设置”入口，支持在程序内配置多个模型供应商、拉取并筛选可用模型、保存到 `backend/.env`，并让聊天页、草稿、后台任务和思考模式一起适配多供应商模型
> 上游背景：[V2.3 Phase 4 文件体验与模型配置实施计划](<../V2.3/V2.3 Phase 4 文件体验与模型配置实施计划.md>) · [V2.4 AI 教练消息附件回显设计](<../V2.4/V2.4 AI 教练消息附件回显设计.md>) · [V2.5 单日训练计划卡采纳设计](<../V2.5/V2.5 单日训练计划卡采纳设计.md>)

## 1. 背景与问题

当前项目的模型配置仍停留在 V2.3 Phase 4 的“单供应商、单配置、手改 `.env`”阶段：

- 后端只支持 DeepSeek
- API Key 和 Base URL 只能手动修改 `backend/.env`
- `/api/models` 只能返回 DeepSeek 的远端模型或本地白名单 fallback
- 前端模型选择器只理解：
  - 一个模型列表
  - 一个全局默认模型
  - 一个 DeepSeek 风格的 `thinking enabled + budget`

这与用户当前要的使用方式不一致。用户要的是更接近常见大模型网页端的体验：

- 在聊天页里直接点“设置”配置模型
- 可以同时保存多个供应商配置
- 不把远端返回的全部模型都丢进下拉框，而是先选出“允许在本产品里使用”的模型
- 模型切换与思考模式切换都在聊天页直接可用
- 配置保存后立即生效，不需要再手工改文件或重启前端心智

本次 V2.6 的核心不是“再加一个表单”，而是把当前单 DeepSeek 配置链路升级为“多供应商模型配置中心 + 聊天运行时 Provider 抽象”。

## 2. 目标

V2.6 目标如下：

- 在 AI 教练页面右上角增加“模型设置”按钮，并提供明确 hover 提示
- 支持在程序内配置模型供应商，而不是要求用户手改 `backend/.env`
- 首期支持：
  - DeepSeek API
  - Google AI Studio Gemini API
- 支持同时保存多个供应商配置
- 每个供应商配置都支持：
  - API Key
  - Base URL
  - 连接测试
  - 拉取远端模型列表
  - 勾选本产品允许展示的模型
- AI 教练模型下拉框只展示“已启用且已勾选”的模型
- 思考模式 UI 能适配不同供应商差异，并通过统一映射返回给前端
- 所有供应商配置最终保存到 `backend/.env`
- 保存配置后，聊天接口、模型列表接口、草稿默认值和后台任务都能读到最新配置

## 3. 非目标

本次不做：

- 云端账户体系或多用户配置隔离
- API Key 加密存储到数据库
- 供应商配置导入导出
- Anthropic、OpenAI、OpenRouter 等更多供应商
- 针对每个模型暴露所有底层高级参数
- 复杂的模型权限系统、团队共享配置、远端同步

## 4. 当前现状梳理

### 4.1 后端配置只支持 DeepSeek 平铺字段

当前 [backend/config.py](/g:/VSCODE-G/Fitness%20Agent%20MVP/backend/config.py) 中与模型相关的配置只有：

- `deepseek_api_key`
- `deepseek_base_url`
- `default_model`
- `model_allowlist`
- `default_thinking_enabled`
- `default_thinking_budget`
- `deepseek_timeout_seconds`

问题：

- 结构是单供应商写死的
- 无法表达“多个供应商、多组 API、多组模型”
- `get_settings()` 使用了 `@lru_cache`，配置写回后不会自动刷新

### 4.2 `/api/models` 只会查询 DeepSeek

[backend/api/models.py](/g:/VSCODE-G/Fitness%20Agent%20MVP/backend/api/models.py) 当前行为：

- 只实例化 `DeepSeekClient`
- 有 Key 时请求 DeepSeek `/models`
- 无 Key 或远端失败时回退到 `MODEL_ALLOWLIST`
- thinking 配置是全局返回，不区分模型与供应商

问题：

- 无法返回多供应商“已启用模型集合”
- 无法表达每个模型自己的能力差异

### 4.3 聊天链路与 Tool Calling 绑定在 DeepSeek 语义上

当前以下模块都带有明显的 DeepSeek 绑定：

- [backend/agent/deepseek_client.py](/g:/VSCODE-G/Fitness%20Agent%20MVP/backend/agent/deepseek_client.py)
- [backend/api/chat.py](/g:/VSCODE-G/Fitness%20Agent%20MVP/backend/api/chat.py)
- [backend/agent/chat_session.py](/g:/VSCODE-G/Fitness%20Agent%20MVP/backend/agent/chat_session.py)
- [backend/agent/tool_calling.py](/g:/VSCODE-G/Fitness%20Agent%20MVP/backend/agent/tool_calling.py)

具体体现：

- 客户端类名就是 `DeepSeekClient`
- thinking 归一化函数叫 `normalize_deepseek_thinking()`
- tool schema 输出方法叫 `to_deepseek_tool()`
- tool loop 默认假设 OpenAI/DeepSeek 风格的 `tools / tool_calls / tool message`

这意味着 V2.6 如果要真正支持 Gemini，不能只改 `/api/models`，必须把聊天运行时抽象成 provider adapter。

### 4.4 前端模型选择器是单供应商视图

[src/components/coach/ModelSelector.jsx](/g:/VSCODE-G/Fitness%20Agent%20MVP/src/components/coach/ModelSelector.jsx) 当前只支持：

- 一个模型下拉框
- 一个是否支持 thinking 的布尔值
- 开启 thinking 后显示 `auto / max`

问题：

- 无法展示“模型来自哪个供应商”
- 无法适配不同模型的不同 thinking 能力
- 无法与“配置中心里筛选后的模型集合”联动

### 4.5 后台任务和应用启动也缓存了旧默认模型

[backend/main.py](/g:/VSCODE-G/Fitness%20Agent%20MVP/backend/main.py) 在模块加载时就读取了一次 `settings = get_settings()`，并在 `lifespan` 初始化 `BackgroundWorker` 时把 `default_model` 固定传入。

问题：

- 只清 `get_settings()` 缓存还不够
- 保存新配置后，如果后台 worker 不刷新，会出现：
  - `/api/models` 已返回新模型
  - 但后台聊天任务仍在使用旧默认模型

## 5. 官方能力依据

以下依据用于说明“为什么 V2.6 必须做 Provider 抽象，而不是继续往 DeepSeek 分支上打补丁”。

### 5.1 Gemini 官方接口能力

基于 Google AI Studio Gemini 官方文档（2026-06-01 查阅）：

- 鉴权方式：
  - Gemini API 支持通过 `x-goog-api-key` 发送 API Key
- 基础 REST 域名：
  - 官方 REST 端点使用 `https://generativelanguage.googleapis.com`
- 生成接口：
  - 文本/多模态生成使用 `models/{model}:generateContent`
  - 流式生成使用 `models/{model}:streamGenerateContent`
- 模型发现：
  - 可通过 `GET /v1beta/models` 列出模型
  - 返回项中包含模型名、展示名和 `supportedGenerationMethods`
- Thinking：
  - Gemini 官方文档提供 `thinking` / `thinkingConfig` 控制能力
  - 不同模型系列对 thinking 预算或强度控制并不完全相同

这说明：

- Gemini 可以满足“在程序内发现模型并选择部分模型展示”的需求
- 但它的生成接口、思考参数、工具调用语义与当前 DeepSeek 并不完全一致
- 前端不能继续假设所有模型都只需要 `supportsThinking + auto/max`

### 5.2 DeepSeek 与 Gemini 的关键差异

当前系统基于 DeepSeek 的关键假设包括：

- `/chat/completions`
- OpenAI 兼容消息格式
- `tools` / `tool_calls`
- `thinking + reasoning_effort`

Gemini 的关键差异包括：

- `generateContent` 不是 `/chat/completions`
- 请求/响应字段结构不同
- thinking 配置入口不同
- tool/function calling 的 schema 与消息回写结构也不同

结论：

- V2.6 必须新增“供应商适配层”
- 不能把 Gemini 当成“另一个 DeepSeek base_url”

## 6. 方案对比

### 方案 A：继续使用平铺 `.env` 字段，为每个供应商加一套变量

做法：

- 新增 `GEMINI_API_KEY`
- 新增 `GEMINI_BASE_URL`
- 新增 `GEMINI_MODEL_ALLOWLIST`
- 继续保留 `DEFAULT_MODEL`

优点：

- 改动表面上最小

缺点：

- 很快失控
- 无法优雅表达多个供应商的多个配置项
- 无法表达同一供应商多组配置
- 前端设置页与后端配置结构耦合过深

结论：

- 不推荐

### 方案 B：在 `.env` 中新增单一 JSON 配置字段，后端做 Provider Registry 抽象

做法：

- 在 `backend/.env` 中新增一个统一配置字段
- 用 JSON 保存所有供应商配置、已选模型、默认模型和思考默认项
- 后端新增 provider registry、config service、env writer、chat adapter

优点：

- 能完整表达多供应商、多模型、多默认项
- 与页面设置中心天然匹配
- 便于未来继续扩展其他供应商

缺点：

- 后端要做一次结构升级
- 需要处理 `.env` 读写、缓存刷新和兼容迁移

结论：

- 推荐采用

### 方案 C：把供应商配置写进 SQLite，仅在 `.env` 中保留一个 DB 开关

做法：

- 页面配置直接写数据库
- 后端从数据库读取模型配置

优点：

- 数据结构更自然
- 修改方便

缺点：

- 用户明确要求保存到 `backend/.env`
- 当前项目对“环境配置”的认知仍以本地 `.env` 为主
- 会引入“代码配置”和“运行时业务数据”混存的心智分裂

结论：

- 本次不采用

## 7. 推荐方案

推荐采用 **方案 B：`.env` 单一 JSON 配置字段 + Provider Registry + 设置弹窗**。

核心原则：

1. 页面内完成配置，但最终仍落到 `backend/.env`
2. `/api/models` 只返回“用户已经选为可用”的模型，而不是远端所有模型
3. 聊天接口收到的 `model` 不再等价于远端原始模型名，而是本地可解析的 `modelRef`
4. thinking 不再是全局硬编码字段，而是每个模型自己的能力描述
5. 旧 DeepSeek 平铺配置保留兼容读取，但新保存统一写入 JSON 配置

## 8. 前端交互设计

### 8.1 设置入口

在 AI 教练页右上角现有操作区新增“模型设置”按钮：

- 使用更清晰的齿轮或滑杆风格图标
- hover 时显示 tooltip：
  - `模型与供应商设置`
- 点击后打开设置弹窗

这符合常见大模型网页端做法：

- 日常聊天时直接切模型
- 需要配置 API 或选择模型时，再进入设置面板

### 8.2 设置弹窗结构

推荐使用右侧抽屉或中等宽度弹窗，不建议跳转独立页面。

建议分为 3 个区块：

#### A. 供应商配置列表

展示当前已保存的供应商配置卡片：

- 配置名称
- 供应商类型
- 是否启用
- 已选择模型数量
- 连接状态摘要

操作：

- 新增配置
- 编辑配置
- 删除配置
- 启用 / 停用

#### B. 配置编辑区

编辑单个供应商配置：

- 配置名称
- 供应商类型：
  - `DeepSeek`
  - `Gemini (Google AI Studio)`
- API Key
- Base URL
- 连接测试按钮
- 拉取模型按钮

说明：

- Gemini 默认填入官方 REST Base URL
- DeepSeek 默认填入官方 Base URL
- Base URL 允许高级用户改写，便于代理或兼容网关

#### C. 模型筛选区

当连接测试或拉取模型成功后，显示该配置的远端模型列表。

每个模型项展示：

- 远端模型名
- 友好展示名
- 是否支持聊天生成
- thinking 能力摘要

用户可勾选：

- 在 FitLoop 中允许使用该模型
- 该模型是否作为默认模型候选

最终只有被勾选的模型才会进入 AI 教练顶部模型下拉框。

### 8.3 聊天页模型选择框行为

聊天页顶部模型选择框更新为：

- 仅显示启用配置中的已选模型
- 展示文案使用：
  - `配置名称 / 模型名称`
  - 例如：`DeepSeek 主账号 / DeepSeek V4 Flash`
  - 例如：`Gemini 主账号 / Gemini 2.5 Flash`

这样可以避免：

- 两个供应商模型名相近时用户分不清
- 后续支持同一供应商多组配置时冲突

## 9. 配置存储设计

### 9.1 `.env` 新字段

新增统一字段：

```env
MODEL_PROVIDER_CONFIG_JSON={"version":1,"defaultModelRef":"provider_deepseek_main::deepseek-v4-flash","providers":[...]}
```

推荐保留旧字段一段兼容期：

- `DEEPSEEK_API_KEY`
- `DEEPSEEK_BASE_URL`
- `DEFAULT_MODEL`
- `MODEL_ALLOWLIST`
- `DEFAULT_THINKING_ENABLED`
- `DEFAULT_THINKING_BUDGET`

但新设置页保存时，以 `MODEL_PROVIDER_CONFIG_JSON` 为唯一真实来源。

### 9.2 JSON 结构

建议结构如下：

```json
{
  "version": 1,
  "defaultModelRef": "provider_deepseek_main::deepseek-v4-flash",
  "providers": [
    {
      "id": "provider_deepseek_main",
      "type": "deepseek",
      "label": "DeepSeek 主账号",
      "enabled": true,
      "apiKey": "sk-***",
      "baseUrl": "https://api.deepseek.com",
      "timeoutSeconds": 30,
      "selectedModels": [
        {
          "remoteId": "deepseek-v4-flash",
          "label": "DeepSeek V4 Flash",
          "enabled": true,
          "thinkingPreset": {
            "enabledByDefault": false,
            "intensity": "standard"
          }
        }
      ]
    },
    {
      "id": "provider_gemini_main",
      "type": "gemini",
      "label": "Gemini 主账号",
      "enabled": true,
      "apiKey": "AIza***",
      "baseUrl": "https://generativelanguage.googleapis.com/v1beta",
      "timeoutSeconds": 30,
      "selectedModels": [
        {
          "remoteId": "gemini-2.5-flash",
          "label": "Gemini 2.5 Flash",
          "enabled": true,
          "thinkingPreset": {
            "enabledByDefault": true,
            "intensity": "standard"
          }
        }
      ]
    }
 ]
}
```

设计要点：

- 上述示例中的 `apiKey` 为脱敏示意；真实保存到 `backend/.env` 时仍是明文密钥
- `id`
  - 本地配置 id，不等于远端供应商名
- `type`
  - 当前只支持 `deepseek` 与 `gemini`
- `label`
  - 前端展示名，可由用户自定义
- `selectedModels`
  - 只保存用户筛选后的模型
- `defaultModelRef`
  - 指向一个本地模型引用，而不是单纯远端模型名

### 9.3 本地模型引用

统一使用：

```text
{providerId}::{remoteModelId}
```

例如：

- `provider_deepseek_main::deepseek-v4-flash`
- `provider_gemini_main::gemini-2.5-flash`

这样做的原因：

- 相同远端模型名不会冲突
- 草稿、后台任务、默认模型都能稳定引用
- 后端收到 `modelRef` 后可以稳定解析出：
  - 使用哪个 provider config
  - 使用哪个远端模型

### 9.4 密钥回显策略

`GET` 配置接口不返回完整 API Key，只返回：

- `hasApiKey`
- `apiKeyPreview`

例如：

```json
{
  "hasApiKey": true,
  "apiKeyPreview": "AIza...9AbC"
}
```

保存时：

- 若用户未改 API Key，前端不回传完整旧 Key
- 若用户新填或替换，则只在 `PUT` 请求中提交一次明文

原因：

- 本项目仍是本地单用户 MVP，但不应该把完整密钥反复暴露给前端读取接口

## 10. 后端架构设计

### 10.1 新增配置服务层

建议新增一个模型配置服务，而不是继续把逻辑散在 `config.py`、`models.py` 和 `chat.py` 里。

推荐新增职责：

- `ModelProviderConfigService`
  - 读取统一 JSON 配置
  - 兼容旧 DeepSeek 平铺配置
  - 输出标准化 provider config
- `EnvConfigWriter`
  - 负责安全写回 `backend/.env`
  - 保留无关环境变量
  - 使用原子写入
- `ProviderRegistry`
  - 根据 provider type 返回对应 adapter

### 10.2 Provider Adapter 抽象

建议新增统一接口：

- `list_remote_models()`
- `test_connection()`
- `generate_chat()`
- `stream_chat()`
- `describe_thinking_capabilities()`
- `build_tool_schema()`

首期实现：

- `DeepSeekProviderAdapter`
- `GeminiProviderAdapter`

这样可以把：

- 模型发现
- 聊天调用
- 流式输出
- tool calling
- thinking 参数映射

都按供应商隔离开。

### 10.3 配置刷新机制

保存设置后，后端至少要刷新 3 层状态：

#### A. `get_settings()` 缓存

调用：

- `get_settings.cache_clear()`

#### B. `backend.main` 模块级 settings

当前 `backend/main.py` 在模块级缓存了 `settings = get_settings()`，这会导致新配置不会反映到：

- CORS 配置之外的若干默认逻辑
- 后台 worker 初始化路径

V2.6 应改为：

- 避免在模块级永久缓存 settings 对象
- 在真正需要的地方重新读取标准化配置

#### C. `BackgroundWorker`

当前 `BackgroundWorker` 初始化时已经吃入默认模型。

V2.6 保存配置后应：

- 重建 worker
  或
- 让 worker 在每次任务执行时动态读取最新默认模型与 provider config

推荐采用第二种：

- 减少“保存配置后重建全局对象”的副作用
- 后续更利于热更新

## 11. API 设计

### 11.1 新增读取配置接口

```text
GET /api/model-config
```

返回：

- 当前标准化配置
- 每个 provider 的 masked key 信息
- 默认模型
- 支持的供应商类型定义

### 11.2 新增保存配置接口

```text
PUT /api/model-config
```

请求体为完整配置快照。

行为：

- 后端校验结构
- 合并未修改的旧 API Key
- 写回 `backend/.env`
- 刷新配置缓存
- 返回保存后的 masked 配置

### 11.3 新增连接测试接口

```text
POST /api/model-config/providers/test
```

请求体：

- `type`
- `apiKey`
- `baseUrl`

返回：

- `ok`
- `message`
- `latencyMs`
- `providerMeta`

作用：

- 校验 Key 和 URL 是否可用
- 在保存前给用户快速反馈

### 11.4 新增远端模型发现接口

```text
POST /api/model-config/providers/discover-models
```

请求体：

- `type`
- `apiKey`
- `baseUrl`

返回：

- `models`
  - `remoteId`
  - `label`
  - `supportsChat`
  - `thinkingCapability`

设计原则：

- 设置页拉“远端全集”
- 聊天页只拉“本地已启用模型”

### 11.5 改造现有模型列表接口

现有：

```text
GET /api/models
```

V2.6 后语义改为：

- 返回所有启用 provider 中，已被用户选中的模型
- 每个模型项都是可直接用于聊天页下拉框的展示对象

建议返回结构：

```json
{
  "defaultModel": "provider_deepseek_main::deepseek-v4-flash",
  "defaultModelRef": "provider_deepseek_main::deepseek-v4-flash",
  "models": [
    {
      "id": "provider_deepseek_main::deepseek-v4-flash",
      "providerId": "provider_deepseek_main",
      "providerType": "deepseek",
      "providerLabel": "DeepSeek 主账号",
      "remoteModelId": "deepseek-v4-flash",
      "label": "DeepSeek 主账号 / DeepSeek V4 Flash",
      "supportsThinking": true,
      "thinking": {
        "supported": true,
        "canDisable": true,
        "defaultEnabled": false,
        "intensityOptions": [
          { "id": "standard", "label": "标准" },
          { "id": "deep", "label": "深入" }
        ],
        "defaultIntensity": "standard"
      }
    }
  ],
  "warning": ""
}
```

兼容策略：

- `defaultModel`
  - 继续保留，值升级为 `modelRef`
- `defaultModelRef`
  - 作为更明确的新字段一并返回
- `supportsThinking`
  - 继续保留给旧 UI 分支使用
- `thinking`
  - 升级为每个模型自己的能力描述，不再依赖全局顶层字段

## 12. 聊天与草稿链路改造

### 12.1 聊天接口的 `model` 字段升级为 `modelRef`

当前前后端默认把 `model` 当成远端模型名。

V2.6 后需要改成：

- 前端仍传字段名 `model`
- 但值变为 `modelRef`

例如：

```json
{
  "model": "provider_gemini_main::gemini-2.5-flash"
}
```

这样可以减少前端破坏面，同时让后端完成真正的 provider 解析。

### 12.2 草稿默认值与历史消息兼容

当前 `CoachDraft.model` 保存的是纯模型名。

V2.6 后：

- 新保存的 draft 记录保存 `modelRef`
- 读取旧 draft 时：
  - 若是纯 DeepSeek 模型名，则按 legacy 规则映射到迁移生成的默认 DeepSeek provider

### 12.3 后台任务兼容

后台任务提交时也必须携带 `modelRef`。

否则会出现：

- 前台请求使用 Gemini
- 后台任务恢复时却回到 DeepSeek 默认模型

## 13. Thinking 统一映射设计

### 13.1 统一前端能力描述

前端不直接理解 DeepSeek 或 Gemini 的原始 thinking 参数，而只消费一个统一结构：

```json
{
  "supported": true,
  "canDisable": true,
  "defaultEnabled": false,
  "intensityOptions": [
    { "id": "standard", "label": "标准" },
    { "id": "deep", "label": "深入" }
  ],
  "defaultIntensity": "standard"
}
```

这样前端只需要处理：

- 是否显示 thinking 控件
- 是否允许关闭
- 有哪些强度可选

### 13.2 统一前端提交结构

前端继续提交一个简化 thinking payload：

```json
{
  "enabled": true,
  "intensity": "deep"
}
```

后端根据当前 `modelRef` 的 provider adapter 做真实映射。

### 13.3 后端映射原则

#### DeepSeek

- `enabled=false`
  - 映射为关闭 thinking
- `enabled=true + intensity=standard`
  - 映射为当前默认的“标准思考”
- `enabled=true + intensity=deep`
  - 映射为更高思考强度

#### Gemini

Gemini thinking 官方能力存在模型差异，因此适配原则定义为：

- 若该模型支持真正关闭 thinking：
  - `enabled=false` 映射为关闭
- 若该模型不支持真正关闭：
  - UI 不展示“关闭”，或 `canDisable=false`
- `intensity`
  - 由 adapter 映射为该模型对应的官方 `thinkingConfig`

这意味着：

- 前端只关心统一语义
- 供应商差异只停留在后端 adapter

## 14. 兼容迁移策略

### 14.1 旧 `.env` 兼容

若 `MODEL_PROVIDER_CONFIG_JSON` 不存在，则后端启动时按旧字段自动合成一个 DeepSeek 默认配置：

- provider type = `deepseek`
- provider label = `DeepSeek 默认配置`
- selected models = `MODEL_ALLOWLIST`
- default model = `DEFAULT_MODEL`
- default thinking = `DEFAULT_THINKING_*`

这样可以保证：

- 旧用户升级后不会立刻失去聊天能力
- 新设置页可以正常展示一个已迁移的默认配置

### 14.2 保存时切换到新配置源

一旦用户在设置页点击保存：

- 后端把完整配置写入 `MODEL_PROVIDER_CONFIG_JSON`
- 之后所有聊天、模型列表和默认配置都以该字段为准

### 14.3 旧接口字段兼容

短期兼容：

- `/api/chat/*` 的 `model` 字段名称不变
- 旧 draft 中的纯模型名可自动映射
- 旧 `/api/models` 消费方仍能拿到 `id / label / supportsThinking`

但真实含义升级为：

- `id` = `modelRef`

## 17. 风险与待确认项

### 17.1 Gemini Tool Calling 改造范围较大

当前 tool calling 流程明显偏 DeepSeek/OpenAI 兼容格式。

风险：

- 如果 Gemini adapter 只先支持基础聊天，不支持 proposal tool calling
- 那么 AI 教练的“采纳建议卡”闭环会退化

因此 V2.6 实现时需要明确：

- Gemini 首期是否要完整支持 tool calling
- 还是先支持基础聊天 + 文件上下文，再在后续版本补 proposal 工具

从产品完整性角度，推荐首期就把 proposal 工具链一起适配进去。

### 17.2 `.env` 存 JSON 的可维护性一般

缺点：

- 人工查看不够友好
- 手动编辑时容易破坏 JSON

但考虑到用户明确要求“保存到 `backend/.env`”，当前仍是最合适方案。

可通过以下方式降低风险：

- 设置页保存时始终由后端生成合法 JSON
- `.env.example` 给出最小示例
- 后端读取失败时返回明确错误，而不是静默回退

### 17.3 配置保存后是否需要重启服务

推荐目标是：

- 不需要用户手动重启

但前提是：

- settings 缓存刷新
- provider registry 重新读取
- 后台任务默认模型不再写死

如果这三项没有一起落地，就会出现“看起来保存成功，实际上运行时还在用旧配置”的假成功问题。

## 18. 结论

V2.6 的本质不是“把 `.env` 改成页面表单”，而是把当前单 DeepSeek 的模型配置与聊天运行时，升级为一个真正可扩展的多供应商模型中心。

推荐落地路径是：

1. 在 `backend/.env` 中新增统一 JSON 配置字段
2. 后端增加 provider config service、env writer 和 provider adapter registry
3. 设置页负责：
   - 配置供应商
   - 测试连接
   - 发现模型
   - 勾选允许使用的模型
4. `/api/models` 只返回本地启用模型集合
5. 聊天与后台任务改为基于 `modelRef` 解析 provider
6. thinking 由后端按模型能力统一映射给前端

这样既符合用户想要的常见网页端模型设置体验，也为后续继续扩展更多模型供应商保留了干净的结构空间。
