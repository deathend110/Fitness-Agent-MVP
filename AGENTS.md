# AGENTS.md

## 项目定位

本项目是 FitLoop MVP：一个本地运行的 AI 健身教练 + 训练记录软件，用于课程设计中的 Agent-based MVP 展示。项目目标是跑通一条最小但完整的核心功能链路：

用户档案 -> 训练计划 -> 今日日志 -> AI 教练上下文注入 -> AI 建议 -> 一键采纳并写回训练计划。

当前阶段优先保证 MVP 可运行、可演示、可验证，避免一次性实现过大的完整系统。

## 开发流程

项目开发必须遵守以下顺序：

1. plan
2. code
3. review

任何较大的功能变更都应先明确计划，再编码，最后进行代码审查和验证记录。

## Agent 协作流程

- 所有“写代码”的任务默认先调用相关的 superpowers workflow skill。
- 代码实现阶段默认使用 `superpowers:subagent-driven-development`，由子代理执行具体编码任务。
- 主 agent 负责整体计划规划、任务拆分、上下文整理、选择合适的 superpowers skill、派发子代理任务，以及检查子代理结果。
- 主 agent 不直接承担默认代码实现工作，而是负责两轮检查：先检查是否符合计划和规格，再检查代码质量、验证结果和文档同步情况。
- 子代理返回的代码结果未经主 agent 审查和验证，不视为任务完成。
- 每次完成一个子任务，或本轮产生实际代码修改并通过对应验证后，必须立即执行一次 git commit。
- 每次上述 git commit 都必须附带**合适的中文说明**，准确描述本次子任务或本次代码修改的核心内容。
- 如果用户明确要求主 agent 直接编码，或者子代理流程因上下文/工具/环境问题被阻塞，主 agent 可以接管实现，但需要在回复中说明原因。
- 文档编写、计划整理、需求分析、代码评审这类非编码任务可以由主 agent 直接完成。

## 编码规范

- 整体代码使用 UTF-8 编码。
- 整体代码需要中文注释。
- 中文注释应解释业务意图、关键数据流、边界情况和非显而易见的实现原因。
- 不写无意义注释，例如“设置变量”“调用函数”。
- 优先使用清晰命名和小函数表达逻辑。
- 保持模块高内聚、低耦合。
- 当前项目是最简 MVP，但后续会继续开发，因此实现时要保留合理扩展性。

## 模块拆分规则

- 不要一次实现过于庞大的模块。
- 单个代码文件超过 200 行时，必须评估是否拆分。
- 过大的模块应按功能拆分为更小组件、工具函数或服务模块。
- React 页面组件应优先只负责页面组织和状态协调。
- 可复用 UI、业务计算、localStorage 操作、AI API 调用、Prompt 构建应拆到独立文件。
- 不把多个无关职责塞进同一个组件。

## 文档维护规则

每次更新代码时，必须同步维护以下文档：

- `README.md`
- `ARCHITECTURE.md`

`README.md` 应至少包含：

- 项目简介
- 环境要求
- 安装命令
- 运行命令
- API Key 配置
- 测试命令
- Demo 操作路径

`ARCHITECTURE.md` 应至少包含：

- 当前项目结构
- 核心模块职责
- 数据流说明
- localStorage 数据结构
- AI 调用链路
- 后续扩展方向

## 技术边界

- 前端使用 Vite + React + Tailwind CSS。
- 数据优先存储在 localStorage。
- AI 接口使用 DeepSeek 原生 OpenAI 格式。
- `.env` 用于保存 `VITE_DEEPSEEK_API_KEY`，不得提交真实 API Key。
- 不在 MVP 阶段引入后端、数据库、账号系统、云同步、复杂食物数据库或社交功能。

## MVP 优先级

优先完成以下闭环：

1. 用户填写档案和三大项 1RM。
2. 用户维护一周训练计划。
3. 用户录入今日日志，包括体重、热量、疲劳度、睡眠和训练备注。
4. AI 教练每次发送消息前自动注入最新上下文。
5. AI 返回训练建议。
6. 如果 AI 回复包含结构化 JSON 建议，页面渲染采纳卡片。
7. 用户点击采纳后，训练计划被更新。

加分项如体重趋势图、流式输出、热力图、数据导入导出，应在核心闭环稳定后再实现。

## 验证要求

每次完成关键功能后，需要至少验证：

- 一条核心成功路径。
- 两到三个边界情况。
- 一个失败场景。

验证结果应能用于课程报告中的 TDD 或最小验证方案，包括测试记录、手动验收清单、截图或日志。

## 协作要求

- 不随意删除用户已有文档和内容。
- 修改已有文件前先理解当前结构。
- 遇到需求冲突时，以课程 MVP 要求和当前 SDD 文档为准。
- 后续实现应持续对齐 `docs/idea.md`、`docs/plan.md`、`docs/fitness_coach_mvp_spec.md` 和课程要求文档。
